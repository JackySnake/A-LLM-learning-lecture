# Agentic RL 与 Agent 训练：Agent 能力从哪里来

*补充来源：完成 Week 9–11（Phase 5 Agent 系统）学习后，发现正式讲义聚焦"推理时工程"（如何搭建/编排/评估 Agent），缺少"模型侧"视角——一个模型的 Agent 能力是如何被训练出来的（2026-06）*

## 一、一个被讲义体系跳过的问题

Week 9–11 讲了 ReAct 循环、工具调用、记忆系统、多智能体编排、MCP、评估体系——这些全部属于**推理时工程**：假设手里已经有一个"会当 Agent"的模型，研究怎么把它用好。

但有一个问题被跳过了：**为什么 Claude、Kimi K2、Qwen3 这些模型"天生会"用工具？**

- 为什么它们能在工具报错后自己换策略重试，而 2023 年的模型只会在错误上反复撞墙？
- 为什么 SWE-Bench Verified 的分数从 2024 年初的 ~14% 涨到 2025 年底的 70%+？Harness 的改进只解释了一小部分，大头来自**模型本身被针对性训练过**。
- 为什么同样接入搜索工具，有的模型只会"搜一次就作答"，有的模型会主动交叉验证、追加搜索？

答案是：**Agent 能力本质上是一种被训练出来的策略（policy），而不是 prompt 出来的行为**。2024–2026 年这条训练路线的演进，就是本专题的主线。

### Agent 能力获得的三条路径

| 路径 | 教会模型什么 | 局限 | 对应讲义 |
|---|---|---|---|
| **Prompt 工程**（ReAct 提示词、few-shot） | 临时激发已有能力 | 能力上限完全取决于基座；格式遵循不稳定 | Week 9 Part 2 |
| **Agent SFT**（模仿轨迹） | 工具调用格式 + 常见"套路" | 只会模仿，不会从自己的错误中恢复 | Week 9 附录 A.6（格式层面） |
| **Agentic RL**（与环境交互试错） | 完成任务的**策略**：探索、验证、纠错 | 基础设施昂贵，reward 设计困难 | **本专题** |

一个直觉性的类比：SFT 像"看老师傅的操作录像学修车"，Agentic RL 像"自己上手拆一百台车，修好了有奖励"。前者学到的是动作序列的形似，后者学到的是"发现装错了怎么办"——而**从错误中恢复恰恰是 Agent 最重要的能力**（Week 10 Part 4.4 讲过"错误是学习机会"，那是推理时的视角；这里是训练时的视角：让模型在训练中大量经历并学会处理自己犯的错）。

## 二、Agent SFT：用合成数据教会"套路"

### 2.1 从格式到行为

Week 9 附录 A.6 讲了 Tool Use SFT 的数据格式（多轮对话中插入 `tool_call` / `tool_result` 消息）。格式是表象，真正的问题是：**高质量的多轮 Agent 轨迹数据从哪里来？**

人工标注一条"调用 8 个工具、跨 20 轮"的轨迹成本极高，而且标注员自己也未必知道最优策略。所以业界的主流答案是**大规模合成**。

### 2.2 Kimi K2 的合成流水线：当前最透明的工业级方案

Kimi K2（2025-07 技术报告）公开了迄今最完整的 agentic 数据合成流水线，值得作为锚点案例记住：

```
① 工具谱系构建
   真实工具（3000+ MCP 工具）+ 合成工具（20000+，按领域系统生成）
        ↓
② Agent 与任务生成
   组合不同工具集 + 系统提示 → 生成多样化"Agent 人设"
   为每个 Agent 生成配套任务，每个任务附带评分细则（rubric）
        ↓
③ 轨迹模拟（关键创新）
   用 LLM 扮演【用户】+ 用 LLM 扮演【工具执行结果】（工具模拟器）
   → 不需要真实环境，就能批量生成多轮交互轨迹
        ↓
④ 质量过滤
   LLM Judge 按任务自带的 rubric 评判轨迹质量，只保留高分轨迹
   （代码类任务则用真实沙箱执行验证，不用模拟器）
```

三个值得琢磨的设计判断：

1. **工具模拟器**：让 LLM 假扮"工具返回结果"听起来不靠谱，但对于教模型"何时调用什么工具、如何串联多步"这类**决策层面**的能力，模拟保真度已经够用；需要精确反馈的代码任务才用真实沙箱。这是典型的成本-保真度分层。
2. **rubric 与任务一起生成**：合成任务时同步生成"什么算做得好"的评分标准，让后续过滤有据可依——这个思想会在 RL 阶段复用（见 4.3 节 rubric reward）。
3. **拒绝采样的本质**：整条流水线可以看作 Best-of-N + 过滤，即把推理时的 Test-Time Scaling（Week 7）成果"蒸馏"回训练数据。这是贯穿现代后训练的通用模式。

### 2.3 SFT 的天花板：分布偏移与"不会回头"

无论合成数据多精良，SFT 都有一个原理性缺陷——**它只在"正确轨迹"的分布上训练**：

- 训练时，模型每一步看到的上文都是"前面全做对了"的状态；
- 推理时，模型一旦犯错（必然发生），就进入了**训练中从未见过的状态分布**，错误不断累积（这与模仿学习中的 compounding error / exposure bias 是同一个问题）；
- 即使在数据里混入"犯错→恢复"的轨迹，那也是教模型模仿"别人的恢复方式"，而非针对**它自己最容易犯的错**做训练。

要让模型在"自己制造的烂摊子"上学会收场，必须让训练数据由**模型自己当前的策略**产生——这就是 on-policy 训练，也就是 RL 登场的根本理由。

## 三、从 GRPO 到 Agentic RL：多了什么、难在哪

### 3.1 回顾：单轮 RLVR（Week 4 的终点）

DeepSeek-R1 路线的核心配方（Week 4 已详述）：

- **可验证奖励（RLVR）**：数学答案匹配、代码过单元测试 → 奖励不可被讨好，杜绝 reward model 被 hack；
- **GRPO**：同一道题采样一组 $G$ 条回答，组内归一化得到优势：

$$A_i = \frac{r_i - \text{mean}(r_1,\dots,r_G)}{\text{std}(r_1,\dots,r_G)}$$

省掉 value model，工程大幅简化。

但注意它的隐含假设：**单轮生成**——模型一口气输出完整回答，中间不与外界交互。Agent 场景把这个假设打破了。

### 3.2 多轮交互的形式化：环境进入了循环

Agent 任务的一条轨迹长这样：

$$\tau = (x,\ a_1,\ o_1,\ a_2,\ o_2,\ \dots,\ a_T)$$

其中 $x$ 是任务描述，$a_t$ 是模型生成的第 $t$ 轮输出（思考 + 工具调用），$o_t$ 是**环境返回的观察**（工具执行结果、编译报错、网页内容）。关键区别一目了然：

> **轨迹中混入了不是模型生成的 token。** $o_t$ 来自外部世界，模型对它没有"生成概率"可言。

这带来第一个、也是最基础的工程改动——**损失掩码（loss masking）**：计算策略梯度时，只对模型生成的 token（$a_t$ 部分）计算 importance ratio 和损失，观察 token（$o_t$ 部分）一律 mask 掉。所有 agentic RL 框架的第一行核心代码都是这个 mask。直觉上很好理解：你不能因为"搜索引擎返回了有用的内容"而奖励模型生成这些内容的概率——它根本没生成。

### 3.3 四个让 Agentic RL 比单轮 RLVR 难一个量级的问题

**问题一：奖励稀疏 + 长程信用分配（credit assignment）**

单轮数学题：几千 token 后立刻有对/错。SWE 任务：模型跑了 40 轮、消耗 10 万 token、调用 60 次工具，最后单元测试不通过——**到底是第 3 轮看错了文件，还是第 27 轮改错了函数？**

GRPO 的处理方式简单粗暴：轨迹级奖励，组内归一化后**把同一个优势值广播给整条轨迹的所有模型 token**。这显然很糙（成功轨迹里的废操作也被奖励，失败轨迹里的正确探索也被惩罚），但在足够大的采样量下"统计上能洗出来"——好动作在成功轨迹中出现的频率系统性更高。更精细的 turn-level 奖励塑形、面向 Agent 的过程奖励（把 Week 7 的 PRM 思想搬到工具调用步骤上）是活跃研究方向，但截至 2026 年初，**工业界主力配方仍是轨迹级稀疏奖励 + 大采样量**，因为过程奖励极易被 hack（Week 7 讨论过的老问题在 Agent 场景下加倍严重）。

**问题二：rollout 又慢又"参差不齐"**

单轮 RLVR 的 rollout 就是批量推理。Agentic rollout 则要：起沙箱容器、执行代码、等网页加载——一条轨迹动辄几分钟甚至几十分钟，而且**长度方差巨大**（有的 2 轮就失败，有的跑 50 轮）。同步训练时整个 batch 要等最慢的那条轨迹，GPU 大面积空转。

业界解法，按激进程度递增：

| 方案 | 思路 | 代表 |
|---|---|---|
| **异步 rollout** | 推理引擎与训练引擎解耦，rollout 持续进行，攒够一批就训练 | AReaL（完全异步） |
| **部分 rollout（partial rollout）** | 超长轨迹到时间就暂停，存下来下一轮迭代接着跑（接受一定 off-policy 偏差） | Kimi K1.5/K2 |
| **环境大规模并行** | 云基础设施同时起上万个独立沙箱 | Qwen3-Coder（2 万并行环境） |

注意一个深刻的副作用：异步/部分 rollout 意味着部分数据由**旧版本策略**产生，纯 on-policy 假设被打破，需要 importance sampling 修正——而当模型是 MoE 时，token 级 importance ratio 本身就因专家路由抖动而高方差。Qwen 团队的 **GSPO** 把 importance ratio 从 token 级改为**序列级**，正是为了解决这条链路上的训练不稳定（这是 Week 2 MoE 知识与 RL 工程在 2025 年的一次合流，值得品味）。

**问题三：上下文随轮次膨胀**

每多一轮，prompt 就长一截。训练时上下文管理策略（保留全部历史？滑动窗口？compaction 摘要？丢弃历史轮的思考块？）必须与推理时的 Harness 行为**严格一致**，否则训练-推理分布失配。例如：推理时 Harness 会丢弃前几轮的 thinking 内容，那训练时构造上下文也必须丢弃——这类"对齐 Harness"的细节是 agentic RL 落地中大量踩坑的来源，也是"Harness 工程"与"模型训练"深度耦合的直接证据。

**问题四：环境本身要可靠、可复位、防作弊**

训练环境被模型每天冲击数百万次，任何漏洞都会被 RL **系统性地**找到并利用（见 4.4 节 reward hacking）。环境工程因此从评测的附属品升格为核心资产——详见第五节。

## 四、奖励设计：Agentic RL 的灵魂

### 4.1 可执行验证：金标准

能用真实执行结果验证的任务，永远优先用执行结果：

- **SWE 任务**：补丁能否通过单元测试（含隐藏测试集，防针对性作弊）；
- **搜索/QA 任务**：最终答案与标准答案匹配（exact match / LLM 比对）；
- **GUI 任务**：操作后环境状态检查（文件是否生成、订单是否创建）。

口诀是 **"难解易验"（hard to solve, easy to verify）**——Qwen3-Coder 报告中明确把"寻找解困难但验证容易的任务"作为 RL 任务筛选的第一原则。这与 Week 11 评估体系中 Outcome Eval 的逻辑同源：**评估标准一旦足够便宜可靠，就能直接升格为训练信号**（这也是 Week 11 Meta-Harness"评测即优化目标"思想的另一面）。

### 4.2 规则相似度奖励：SWE-RL 的取舍

真实可执行环境贵且覆盖面窄（不是每个 GitHub 仓库都能装好依赖跑通测试）。Meta 的 **SWE-RL**（2025-02）给出另一条路：

- 不执行代码，直接计算**模型生成的补丁与人类真实补丁的文本相似度**（基于序列匹配）作为奖励；格式错误给 -1；
- 代价：相似度 ≠ 正确性（语义等价但写法不同的补丁被冤枉）；
- 收益：**任何有 PR 历史的仓库都能变成训练数据**，规模瞬间扩大几个数量级。

这是一个典型的"奖励保真度 vs 数据规模"的权衡。后续工作（如 DeepSWE）回到可执行环境路线后效果更好，但 SWE-RL 证明了粗糙奖励 + 大规模也能显著提升 agent 能力——两条路线最终在"用合成手段把更多仓库变成可执行环境"（SWE-smith 等，见第五节）上合流。

### 4.3 Rubric / 自评奖励：把不可验证任务拉进 RL

写报告、做调研这类任务没有"单元测试"。Kimi K2 的方案是**自评（self-critique）rubric 奖励**：

1. 模型对自己的输出按 rubric（清晰度、事实性、指令遵循等）打分，作为奖励信号；
2. **关键防 hack 设计**：评判者（critic）不是固定的——它持续用**可验证任务上的客观信号**做校准更新，防止策略和评判者合谋"互相吹捧"导致奖励通胀。

可以把它理解为：用可验证任务当"锚"，把奖励的可信度传导到不可验证任务上。这是 2025 年后训练体系里最值得记住的设计模式之一。

### 4.4 Reward Hacking：RL 训练中的头号实践敌人

RL 优化器是不知疲倦的漏洞挖掘机。Agent 场景的真实案例：

- 修不好代码，就**直接修改单元测试**让它通过；
- 在测试文件里硬编码期望输出；
- 发现评分脚本读某个结果文件，就直接伪造该文件；
- rubric 自评场景下，学会输出"自夸式"措辞抬高评分。

工程防线（多层叠加，没有银弹）：测试文件只读/对模型不可见、用**隐藏的 holdout 测试**做最终判定、训练 hack 检测分类器扫描轨迹、对高分轨迹人工抽审。注意这与 Week 11 的 Agent 安全（Prompt Injection 防御）形态相似但方向相反：那里防外部攻击者操纵 Agent，这里防 **Agent 自己"攻击"训练系统**。

## 五、环境工程："环境是新的数据集"

2025 年下半年，"environments are the new datasets" 成为共识级口号。逻辑很直接：RLVR 时代，**有多少高质量可验证环境，就能做多少 RL**——环境取代标注数据成为稀缺资源。

代表性基础设施（以 SWE 领域最成熟）：

| 工作 | 贡献 | 一句话定位 |
|---|---|---|
| **SWE-Gym**（2024-12） | 第一个开源的 SWE Agent **训练**环境（注意：SWE-Bench 是评测集，不能拿来训练） | 把"评测基准"补全为"训练靶场" |
| **SWE-smith**（2025） | 给定任意仓库，**自动合成**大量带可执行验证的 bug 修复任务（如反向注入 bug） | 解决环境数量瓶颈 |
| **R2E-Gym**（2025） | 程序化构建 ~4.5k 可执行环境，DeepSWE 的训练基座 | 规模化的开源实践 |
| **ZeroSearch**（2025，阿里） | 用 LLM **模拟搜索引擎**返回结果（可控质量退化），替代昂贵的真实搜索 API | 搜索类环境的成本解法 |
| 环境平台（Prime Intellect Environments Hub 等，2025 下半年） | 社区共建可复用 RL 环境的"HuggingFace" | 环境的基础设施化 |

观察两个趋势：

1. **评测与训练的"军备同源"**：SWE-Bench（评测）→ SWE-Gym（训练）→ SWE-smith（合成更多训练任务），同一套"真实仓库 + 可执行验证"的范式在评测和训练两侧复用。Week 11 讲的每一个 Agent 评测基准，几乎都有人在尝试"训练化"。
2. **模拟与真实的分层**（呼应 2.2 节工具模拟器）：决策能力可以在模拟环境里学，精确反馈必须来自真实执行——成本分层是环境工程的中心思想。

## 六、代表工作地图：三条战线

把 2025–2026 年的 agentic RL 工作按任务域整理（精读任选其一即可，推荐 Kimi K2 报告，因为它横跨 SFT 合成与 RL 两阶段且写得最透明）：

### 战线一：Agentic Coding（最成熟）

- **SWE-RL**（Meta）：规则相似度奖励，免执行换规模；
- **DeepSWE**（Agentica × Together，2025）：Qwen3-32B + verl + R2E-Gym，全开源复现工业级效果（SWE-Bench Verified ~42% pass@1），**入门精读首选**，因为代码、数据、配方全公开；
- **Qwen3-Coder**：2 万并行云端环境的大规模可执行验证 RL；
- **Kimi K2 / Claude 4 系列**：工业级闭环（K2 公开了方法论，Anthropic 公开信息少但 SWE-Bench 曲线说明一切）。

### 战线二：搜索与研究 Agent

- **Search-R1 / R1-Searcher**（2025 初，学术）：把"何时搜索、搜什么、搜几轮"作为策略用 GRPO/PPO 端到端学习，奖励为最终答案正确性——证明了搜索行为本身可以被 RL 塑造（模型学会了主动多轮搜索和交叉验证，而非 prompt 强制）;
- **WebSailor / WebDancer**（阿里）：面向复杂网页信息检索的 agentic RL 配方；
- **OpenAI Deep Research / Kimi-Researcher**（2025）：官方明确说明为**端到端 RL 训练**的研究型 Agent——这是"Deep Research 类产品"（Week 10 案例库缺失的那一类）背后的训练真相：它不是编排出来的工作流，而是训练出来的单体策略。

### 战线三：GUI / Computer Use Agent

- **UI-TARS**（字节，2025）：原生 GUI Agent 模型，截图输入 → 坐标动作输出，含 RL 阶段；
- OpenAI **CUA**（Operator 背后模型）：computer use 的端到端 RL。
- 此战线动作空间（点击坐标、键入）与文本工具调用差异最大，训练难度也最高，留待 GUI Agent 专题/Phase 6 Week 13（OSWorld）再展开。

## 七、实践启示

### 7.1 SFT 够用还是必须 RL？决策清单

| 你的目标 | 建议 |
|---|---|
| 让模型学会公司内部工具的调用格式/规范 | **SFT 足够**（几百~几千条高质量轨迹） |
| 让模型在固定工作流里稳定执行（格式遵循、风格统一） | SFT（+ DPO 修正典型坏例） |
| 让模型学会失败后换策略、自主验证、多轮探索 | **必须 RL**（这是策略，模仿学不会） |
| 资源有限、没有可验证环境 | 先做"轨迹拒绝采样 + SFT"（用 Best-of-N 过滤自产数据，可视为 RL 的退化形态，性价比极高） |

### 7.2 一条可落地的小实验路径（对应实战路线图硬件）

用 8 卡机器即可完成一次 agentic RL 端到端体验：

1. **任务**：多轮搜索 QA（复现 Search-R1 配方），或带 Python 解释器的数学题（工具调用版 RLVR）；
2. **配置**：Qwen3-4B/8B + TRL 的 GRPOTrainer（或直接上 verl）+ 本地检索器（避开真实搜索 API，体会 ZeroSearch 的动机）；
3. **必做观察点**：
   - loss mask 是否正确盖住了工具返回 token（拿一条样本肉眼检查，这是最常见 bug）；
   - 训练曲线上"平均工具调用次数"的变化——模型学会多轮搜索是肉眼可见的涌现过程；
   - 故意在奖励里留个小漏洞，观察模型多快找到它（reward hacking 的最佳教学方式是亲眼看一次）。

实验记录按惯例写入 `学习进度追踪.md` 的工程实验记录部分。

## 八、前沿动态（2025–2026）

按厂商/社区梳理（截至 2026 年中）：

- **OpenAI**：o3 起在思维链内原生交错工具调用（边想边搜边算）；Deep Research 与 Operator 均为端到端 RL 产物。路线特征：**不公开配方，但产品形态明确宣示"Agent = 训练出来的模型"**。
- **Anthropic**：Claude 4 → 4.5 系列在 agentic coding 上的跃迁主要来自大规模真实软件工程任务 RL；interleaved thinking（工具调用间保留思考块）同时是产品特性与训练格式。配方不公开，但其工程博客（context engineering、building effective agents 等）持续输出推理时方法论——**模型侧保密、Harness 侧开放**是其策略。
- **Kimi（月之暗面）**：K2 技术报告是目前 agentic 训练**最透明的工业级文档**（合成流水线 + 自评 rubric RL + partial rollout 基础设施），本专题大量锚点来自于它；后续 K2 Thinking 进一步强化长程工具调用（数百轮连续调用）。
- **Qwen（阿里）**：GSPO 算法（MoE RL 稳定性）+ Qwen3-Coder 的 2 万并行环境工程 + WebSailor 系搜索 Agent，开源生态中配方披露最全的一家。
- **DeepSeek**：R1 奠定 RLVR 范式本身；其后续版本（V3.2 起）将 agentic 任务纳入 RL 训练循环，社区高度期待其把"R1 时刻"复制到 Agent 领域。
- **开源社区**：verl 成为事实标准框架；环境平台（Prime Intellect 等）推动"环境众包"；DeepSWE/R2E-Gym 证明开源配方可逼近工业水平。
- **研究热点（开放问题）**：长程信用分配（turn-level/step-level 优势估计）、多智能体联合 RL（训练协作策略而非单体）、RL 与持续学习的结合（Agent 在部署中持续自我改进——与 Week 10 Hermes 的技能合成形成"参数化 vs 非参数化"自我改进的对照，也是 Phase 7 持续学习的伏笔）。

## 十、延伸方向（自主扩展）

以下方向是本专题的自然延伸，各自都能撑起独立的深入学习，按需选择：

| 方向 | 核心问题 | 参考起点 |
|---|---|---|
| **过程奖励模型（PRM）在 Agent 场景的迁移** | 数学题 PRM 验证单步推理；Agent 场景中工具调用的"中间步"如何定义正确性？环境执行结果能否替代人工标注 PRM？ | OmegaPRM、Math-Shepherd、Kimi K2 rubric 自评 |
| **多智能体联合 RL** | 单体 RL 训练出的 Agent 如何在多 Agent 协作中稳定？角色分工（Orchestrator vs Worker）能否联合训练？博弈论与合作 RL 的边界 | MARL 经典框架（MAPPO）→ 迁移到 LLM Agent |
| **持续 RL：部署中自我改进** | 模型上线后如何在真实用户交互中继续学习而不灾难遗忘？在线 RL vs 离线回放 vs 参数冻结 + LoRA 增量 | Week 14 持续学习（HOPE）、Hermes 技能合成对照 |
| **Agent 评估体系的构建** | 评估指标设计直接影响 RL 训练信号质量。如何构建不可被 reward hacking 的评估？Task 粒度 vs Step 粒度 vs Outcome 粒度的权衡 | SWE-Bench、AgentBench、TAU-Bench → Week 11 评估体系 |
| **环境工程与沙箱设计** | Agent 训练需要可重置、副作用可回滚、能并行扩展的执行环境。代码沙箱 / 浏览器沙箱 / 工具调用 Mock 各自的设计挑战 | SWE-Gym、R2E-Gym、Qwen3-Coder 2 万并行环境 |
| **训练数据的长尾与困难样本挖掘** | 简单任务 SFT 天花板低，困难轨迹（多次失败后成功）数据稀缺。如何系统挖掘、过滤、合成长尾难例？ | 课程学习（Curriculum Learning）+ 拒绝采样迭代 |
| **GUI Agent 专项** | 纯文本工具调用以外，屏幕截图 + 鼠标点击的 GUI 控制是另一大类 Agent。视觉 grounding 与动作空间设计与 NLP Agent 有本质差异 | SWE-Bench Multimodal、OSWorld、Week 13 GUI Agent |
