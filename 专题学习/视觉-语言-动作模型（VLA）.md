## 一、VLA 要解决什么问题

**机器人控制的终极挑战**：让机器人能理解任意自然语言指令（"帮我把红色的杯子放到托盘上"），看到摄像头画面后，输出精确的控制信号让机械臂执行。

这件事难在哪？它要求模型**同时**做三件性质完全不同的事：

1. **看懂场景**（视觉语义理解）：红色的杯子在哪里？托盘在哪里？两者的相对位置是什么？
2. **理解指令**（语言理解）："放到托盘上"意味着什么操作序列？
3. **输出动作**（运动控制）：机械臂各关节角度如何变化，末端执行器走什么轨迹，夹爪什么时候闭合？

三者所需的知识深度截然不同——视觉语义需要海量图片训练，运动控制需要精确的物理感知，语言理解需要广泛的世界知识。传统方案把这三个模块分开设计，而 VLA 的目标是**用一个统一的模型端到端地解决所有三件事**。

## 二、前 VLA 时代：从行为克隆到 RT-1

理解 VLA 为什么出现，需要先看它的前身做了什么，以及遇到了哪些无法绕过的墙。

### 2.1 行为克隆（BC）：最直觉的起点

**做法**：给机器人收集大量"专家演示"——人类控制机械臂完成任务，同时记录每一帧的图像和对应的控制指令，然后训练一个神经网络学习"图像 → 动作"的映射。这个方法叫**行为克隆（Behavioral Cloning, BC）**，本质是监督学习。

**核心缺陷**：

- **分布偏移（Distribution Shift）**：训练时模型只见过专家演示的状态，一旦执行过程中出现任何偏差（比如机械臂的位置稍有误差），就进入了模型从未见过的状态分布，错误迅速累积直至任务失败。
- **数据量与任务强绑定**：每个新任务都需要从头收集演示、重新训练，没有跨任务迁移能力。
- **不懂语言**：无法接受自然语言指令——你需要为"放杯子"和"拿杯子"分别训练两个不同的策略。

### 2.2 BC-Z 和 RT-1：走向泛化的过渡

**BC-Z**（Google，2021）是第一批系统性研究跨任务泛化的工作。做法是：

1. 用一个语言编码器把任务描述编码为向量
2. 把这个向量与图像特征拼接，输入策略网络
3. 在数百个任务上联合训练

BC-Z 证明了：**增加任务多样性**（而非单个任务的数据量）是提升泛化的关键。但它的视觉骨干和语言骨干都是专门为机器人任务从头训练的，没有利用互联网预训练知识，泛化边界还是有限。

**RT-1**（Robotics Transformer 1，Google，2022）迈出更重要的一步：使用**Transformer 架构**替代 CNN，并在一个覆盖 700+ 任务、130,000+ 真实机器人演示的大规模数据集上训练。RT-1 证明了机器人领域的"数据规模定律"：足够多的任务多样性加上 Transformer 的序列建模能力，可以让策略学会对新任务的零样本/少样本泛化。

但 RT-1 仍然是从零训练的——视觉和语言都没有利用互联网规模的预训练。这意味着它学到的"世界知识"仍然局限于机器人演示数据集本身。

### 2.3 关键洞见：互联网规模的知识库已经存在

就在 RT-1 训练的那段时间，视觉语言模型（VLM）领域发生了翻天覆地的变化——CLIP、Flamingo、GPT-4V 等模型已经从海量图文数据中学到了惊人的视觉语义理解能力：它们知道"杯子"长什么样，理解"把 A 放到 B 上面"意味着什么，甚至能做基本的空间推理。

**关键问题**：能否把这些"白得"的能力嫁接到机器人控制上，而不是在机器人数据上从头训练？

## 三、VLA 的核心思想：预训练 VLM 作为机器人的"大脑"

VLA（Vision-Language-Action Model）的核心思想：

> **在已经预训练好的大型视觉语言模型（VLM）基础上，增加一个动作输出头，用少量机器人演示数据做微调，把 VLM 的视觉和语言理解能力迁移到动作输出任务上。**

好处是多方面的：

- **视觉和语言理解"免费"**：不需要从机器人数据重学"杯子是什么"，VLM 已经知道了
- **指令跟随能力**：直接继承 LLM 的自然语言理解能力，可以处理从未见过的任务描述
- **数据效率大幅提升**：泛化能力来自 VLM 的预训练，机器人演示数据只需教它"如何动"
- **常识推理**：遇到意外情况时（物体倒了、位置偏移），VLM 的常识知识能帮助推断正确响应

**RT-2**（Google，2023）是第一个规模化验证这个想法的工作——直接用 VLM（PaLI-X 和 PaLM-E）作为骨干，在机器人数据上微调后获得了前所未有的跨任务泛化能力，由此正式开启了 VLA 时代。

## 四、VLA 基础架构：在 VLM 三大件上增加动作头

> **📖 关联讲义：Week 8 Part 1** 详细介绍了多模态 VLM 的"三大件"架构（视觉编码器 → 跨模态投影层 → LLM 骨干）。VLA 在这个基础上增加第四个组件：**动作解码器**。

### 4.1 完整架构图

```
输入：摄像头图像（单帧或多帧）+ 语言指令文本
              │                        │
              ▼                        ▼
  ┌──────────────────────┐   ┌──────────────────────┐
  │  视觉编码器            │   │  文本 Tokenizer       │
  │ (ViT / SigLIP)       │   │  (BPE 分词)           │
  └──────────┬───────────┘   └──────────┬───────────┘
             │ 图像特征向量               │ 文本 Token IDs
             ▼                          ▼
  ┌──────────────────────┐   ┌───────────────────────┐
  │  跨模态投影层          │   │  文本 Embedding 层─    │
  │  (MLP Projector)     │   │                       │
  └──────────┬───────────┘   └───────────┬───────────┘
             │ 图像 Token（视觉 embedding）│ 文本 embedding
             └──────────┬────────────────┘
                        │ 拼接为统一 Token 序列：
                        │ [IMG_START] <视觉token×N> [IMG_END] <文本token×M>
                        ▼
           ┌─────────────────────────────┐
           │      LLM 骨干                │
           │  (Transformer，因果注意力)    │
           └──────────────┬──────────────┘
                          │ 隐状态序列
                ┌─────────┴──────────┐
                ▼                    ▼
     ┌──────────────────┐  ┌────────────────────────┐
     │  文本输出头        │  │   动作解码器            │
     │ (next token预测)  │  │ (Regression / Flow     │
     └──────────────────┘  │  Matching / Diffusion) │
                           └──────────┬─────────────┘
                                      ▼
              输出：动作向量 [关节角度 / 末端位姿 / 夹爪开合度]
```

### 4.2 各组件的角色与选型

#### 视觉编码器（Vision Encoder）

**作用**：把摄像头图像转化为一组高维特征向量（视觉 Token）。

**主流选型**：
- **CLIP ViT-L/14**（OpenAI）：早期 VLM（如 LLaVA、InstructBLIP）的标准配置
- **SigLIP**（Google）：目前更主流的选择。用 Sigmoid 损失替代 CLIP 的 Softmax 对比损失，对大 Batch 更友好，且在下游任务上表现更好

ViT 的核心做法：把图像切成固定大小的 Patch（如 14×14 像素），每个 Patch 转成一个向量，通过多层 Transformer Self-Attention 提取层级语义特征（像素 → 边缘 → 纹理 → 物体 → 场景）。

> **📖 关联讲义：Week 8，第一节"视觉编码器"** 详细讨论了 CLIP vs SigLIP 的技术差异，以及为什么 ViT 比 CNN 更适合与语言模型结合。

**VLA 的特殊需求**：机器人任务的视觉输入往往是**多摄像头、多帧**的（例如：腕部摄像头 + 全局摄像头，当前帧 + 历史帧）。因此部分 VLA 架构会对视觉编码器做适配，如时序 patch 池化（temporal patch pooling）来压缩帧数。（多摄像头与多帧输入的完整设计见**附录 A.3**）

#### 跨模态投影层（Projector）

**作用**：将视觉编码器输出的特征维度，对齐到 LLM 的 hidden size，使图像特征能作为 Token 参与 Transformer 的注意力计算。

**主流选型**：一个**2层 MLP**。早期的 Q-Former（BLIP-2 提出）虽然能进一步压缩视觉 Token 数量，但 MLP Projector 在精度上往往更好，现在几乎是默认配置。

MLP Projector 会引入一定的**信息压缩**——ViT 输出的每个 patch 特征（如 1024 维）被线性投影到 LLM hidden size（如 4096 维），几何上是维度扩张，但多个 patch token 有时会被池化合并以减少序列长度。

> **📖 关联讲义：Week 8，附录 A.6** 详细比较了 Q-Former、Perceiver Resampler 和 MLP Projector 三种投影方式的演进历程与工程取舍。

#### LLM 骨干（LLM Backbone）

**作用**：接收"视觉 Token + 文本 Token"拼接的统一序列，通过多层 Transformer 的 Self-Attention 实现跨模态信息融合，输出包含场景语义的隐状态序列。

**格式细节**：图像和文本以特殊 Token 分隔后拼接：

```
序列格式：<IMG_START> [视觉 Token × 256] <IMG_END> "把红色杯子放到托盘上" <|endoftext|>
```

LLM 骨干使用**因果注意力掩码（Causal Mask）**——每个位置只能看到左侧的 Token，图像 Token 通常被允许互相双向可见（因为它们是静态的，没有时序因果约束）。

**VLA 中 LLM 的权重来源**：直接复用预训练好的 LLM（如 LLaMA-3-8B、Qwen2.5-7B）权重。这是 VLA 与纯行为克隆方案的本质区别——**"理解世界"的能力免费继承，只需要教它"动起来"**。

#### 动作解码器（Action Decoder）—— VLA 专有

**作用**：把 LLM 最后一层的隐状态特征，转化为机器人可执行的连续动作向量。这是 VLA 与纯 VLM 唯一的结构性差异，也是技术路线分化的核心。

动作解码器有三种主流设计，详见第六节。

## 五、动作表示：如何把"动作"变成数字

机器人的一个动作通常表示为一个**低维连续向量**，各维度均有明确的物理意义：

| 分量 | 含义 | 典型范围 |
|------|------|---------|
| **关节角度**（Joint Angles） | 每个关节的旋转量（弧度或角度） | 由机器人型号决定 |
| **末端执行器位姿**（EEF Pose） | 抓手的 XYZ 位置（3维）+ 旋转四元数（4维） | 工作空间内 |
| **夹爪指令**（Gripper） | 0（张开）到 1（完全闭合） | [0, 1] |

典型动作维度：

- **单臂机器人**（如 Franka Panda）：7 轴 + 1 夹爪 = **8 维**
- **双臂机器人**（如 Aloha）：**14–16 维**
- **人形机器人**（如 Figure、Unitree G1）：**50+ 维**

> **💡 连续 vs 离散动作**
> 动作的每个维度是连续实数（如关节角 = 47.382°），这对机械臂精细操控是必须的。但语言模型天生只能预测离散 Token（词汇表中的整数 ID）。这个根本矛盾催生了动作解码器的不同设计路线（见第六节）：要么把连续动作**离散化成 Token**让 LLM 生成，要么在 LLM 上挂一个**独立的连续动作生成器**（扩散/Flow Matching）。

### 5.1 归一化

不同机器人的关节范围差异极大（有的是 0°~360°，有的是 -180°~180°）。训练前通常对每个维度做**分位数归一化**（Quantile Normalization）——把各维度的数值缩放到统一范围（如 [-1, 1]），防止量级大的维度主导损失，同时让模型学到的表示在不同机器人间更容易迁移。

### 5.2 与 Embedding 向量的区别

动作向量与神经网络中的 Embedding（嵌入向量）容易混淆：

| 维度 | 动作向量 | Embedding 向量 |
|------|---------|---------------|
| **每个维度含义** | 明确、物理可解释（第3维 = 肩关节角度） | 无明确语义，是模型学出来的隐式表示 |
| **数值来源** | 直接从传感器/控制器读取或预测 | 神经网络参数，通过反向传播优化 |
| **典型维度数** | 8–56（由机器人 DoF 决定） | 512–4096（由模型设计决定） |
| **可解释性** | 高（每维直接对应物理量） | 低（整体向量才有意义） |

## 六、动作解码器：三种技术路线

动作解码器的设计是 VLA 流派分化的核心，三种路线对应不同的精度-速度取舍。

### 6.1 直接回归（Direct Regression / Token 离散化）

**代表**：RT-2、OpenVLA

**做法**：把连续动作向量的每个维度量化成 256 个离散 Bin（类似数值量化），当作特殊 Token 加入词表，让 LLM 像预测文字一样逐个 Token 预测动作。

```
连续关节角 0.73 → 量化为 Bin #187 → Token "ACT_187"
```

训练损失：交叉熵（与文本生成完全相同的目标函数）

**优缺点**：

| | 说明 |
|--|--|
| ✅ 架构极简 | 直接复用 LLM 的自回归生成能力，无需任何额外组件 |
| ✅ 多任务联合训练自然 | 动作 Token 和文本 Token 在同一个词表中，可以无缝混合 |
| ❌ 精度损失 | 量化精度有限，256 Bin 对某些精细操作（如插针、拧螺丝）远不够 |
| ❌ 多峰失效 | 回归目标隐含单峰假设，面对多种合法操作时预测"平均动作"（见 6.4 节） |
| ❌ 推理慢 | 自回归逐 Token 生成，动作维度越多越慢 |

### 6.2 扩散策略（Diffusion Policy）

**代表**：Diffusion Policy（Chi et al., 2023）、π₀（Physical Intelligence）

**做法**：动作解码器是一个扩散模型。以 LLM 最后一层的隐状态作为条件（condition），扩散模型从标准高斯噪声出发，迭代去噪，生成最终的动作向量。

```
噪声 a_T → 去噪 T 步（以 LLM 隐状态为条件） → 干净动作 a_0
```

> **📖 关联专题：[扩散模型](扩散模型.md)** 详细讲解了扩散模型（DDPM、DDIM、LDM）和 Flow Matching 的完整原理。以下只做要点提示。

扩散策略的核心优势在于对**多峰动作分布**的处理（见 6.4 节）。去噪网络（通常是 U-Net 或小型 Transformer）接收含噪动作和条件特征，训练目标是预测每步应该去掉的噪声量：

$$\mathcal{L} = \mathbb{E}_{k, \epsilon} \left\| \epsilon - \epsilon_\theta(a_k, k, z) \right\|^2$$

其中 $z$ 是 LLM 的隐状态，$a_k$ 是第 $k$ 步含噪动作，$\epsilon$ 是真实噪声。

**优缺点**：

| | 说明 |
|--|--|
| ✅ 多峰建模 | 天然支持多峰动作分布，不同采样运行可产生不同的合理轨迹 |
| ✅ 精度高 | 连续空间生成，无量化误差，精细操作效果好 |
| ❌ 推理慢 | 原始 DDPM 需要 100–1000 步去噪，即使 DDIM 也需要 10–50 步 |

### 6.3 Flow Matching（当前主流）

**代表**：π₀.5（Physical Intelligence）、LingBot-VA（蚂蚁集团）

**做法**：用 Flow Matching 替代扩散过程。思路相同（都是从噪声生成动作），但训练目标是学习"从噪声 $a_0$ 到干净动作 $a_1$ 的直线流"而不是曲折的扩散路径：

$$\mathcal{L}_{\text{FM}} = \mathbb{E}_{t, a_0, a_1} \left\| f_\theta(a_t, t, z) - (a_1 - a_0) \right\|^2$$

其中 $a_t = t \cdot a_1 + (1-t) \cdot a_0$（线性插值）。由于路径是直线，推理时 ODE solver 只需 **3–10 步**即可获得高质量动作，比扩散策略快 5–10 倍。

> **📖 关联专题：[扩散模型](扩散模型.md) "Flow Matching"一节** 详细推导了为什么直线路径能用更少步数完成同等质量的生成。

动作预测头通常是一个**小型 MLP**（2-4 层，hidden dim 256–512），接收 LLM 隐状态 $z$、含噪动作 $a_t$、时间步 $t$（正弦编码后输入），输出预测的速度场。选用 MLP 而非 Transformer 的原因：动作维度小（≤56），MLP 一次前向传播生成全部维度（自然建模维度间相关性），推理延迟低，符合实时控制需求。

**优缺点**：

| | 说明 |
|--|--|
| ✅ 精度接近扩散策略 | 保留了多峰建模能力 |
| ✅ 推理快 | 3–10 步即可，满足实时控制（~50Hz）需求 |
| ✅ 训练稳定 | 线性路径梯度方差小，比 DDPM 训练更稳定 |
| ❌ 框架较新 | 2022 年才提出，调参经验积累不如扩散模型丰富 |

### 6.4 为什么主流放弃直接回归，转向扩散/Flow Matching？

理解这个问题的关键是**多峰动作分布（Multimodality）**。

**什么是单峰和多峰分布？**

"分布"描述的是：给定某个输入，输出值落在各个区域的可能性有多大。用概率密度曲线表示时，曲线的"峰"就是"高概率区域"。

- **单峰分布（Unimodal）**：只有一个高概率区域，典型例子是高斯分布（正态分布）——大多数值聚集在均值附近，越偏离均值概率越低。物理含义是"答案基本上就是那一个，偏差只是随机噪声"。
- **多峰分布（Multimodal）**：有多个高概率区域，每个"峰"代表一类同样合理的答案。物理含义是"有好几种本质不同的正确答案，从哪个都行"。

```
单峰分布（e.g. 高斯）        多峰分布（e.g. 双峰）

     ┃                            ┃         ┃
   ████                         ██┃   +   ██┃
 ████████                      ████       ████
─────────────              ────────────────────
  一个正确答案                两种都正确的答案
```

直接回归（MSE/L2 损失）隐含了一个假设：给定当前观测，最优动作服从**单峰高斯分布**。但机器人任务恰恰相反——同一观测下往往存在**多种同样合理的操作方式**（从左侧抓、从右侧抓、从上方抓，都是对的）。当模型用回归拟合多峰数据时，会输出所有峰的**加权平均**，产生一个实际上对哪种方式都不奏效的失效动作。

```
真实动作分布（双峰）：      回归模型输出：
      ┃         ┃                    ┃
      ┃    +    ┃          →         ┃（两峰中间的"空地"）
──────┸─────────┸──────       ──────────────────────
```

| 解码方式 | 分布假设 | 多峰处理 | 精度 | 推理速度 |
|---------|---------|---------|------|---------|
| 直接回归（连续） | 单峰高斯 | 差（均值平均） | 中 | 快 |
| Token 离散化（RT-2） | 无，但有量化误差 | 差 | 低 | 中 |
| 扩散策略 | 任意分布，无假设 | 好（采样不同峰） | 高 | 慢 |
| Flow Matching | 任意分布，无假设 | 好 | 高 | 中（可接受） |

### 6.5 为什么 AR + FM/Diffusion 的混合结构成为标准

6.4 节说清楚了"不用纯回归"的原因。但接下来该用什么？上面的对比表给出了四个选项，而实践中已经形成了一个明显的趋势：**用 LLM（AR）处理上下文，用 Flow Matching 或扩散模型生成动作**。这两件事为什么要分开做，而不是用同一个模型端到端完成？

**AR 和 Diffusion/FM 各自能做什么、不能做什么**：

| | 擅长 | 不擅长 |
|--|------|-------|
| **自回归 LLM（AR）** | 序列语义理解、跨模态注意力、指令理解、长程上下文 | 生成连续值的多峰分布（量化损失精度） |
| **Flow Matching / 扩散** | 连续值的多峰分布建模、任意精度动作生成 | 处理变长的多模态序列上下文 |

两者的能力恰好互补，分工非常自然：

```
视觉 Token + 语言 Token + 历史帧
              ↓
     LLM 骨干（自回归处理）
     "理解当前情况，压缩成隐状态"
              ↓
        隐状态向量 z
              ↓
     FM / 扩散动作头
     "从 z 出发，生成连续动作向量"
     （支持多峰分布，无量化损失）
              ↓
        动作向量 a_t
```

**三个具体例子：**

- **π₀**：PaliGemma（3B VLM）作为 AR 骨干处理多模态上下文，独立的"动作专家（action expert）"用 Flow Matching 生成动作——两部分权重分开，动作专家只负责从骨干隐状态推动作
- **OpenVLA-OFT**：LLaMA 7B 作为 AR 骨干，末尾接 FM 动作头，替换掉原始 OpenVLA 的 Token 离散化解码
- **LingBot-VA**：整体是 AR 视频-动作交织序列，动作 token 本身用 FM 框架生成连续潜在向量而非离散 token

这三个方案都体现同一个设计哲学：**LLM 不必学会生成连续物理量，它只需要生成高质量的条件向量；生成连续量的任务交给专门为此设计的 FM/扩散头。**

这个混合结构也是 WAM 两种范式的共同基础——LingBot-VA 把 AR 层扩展到了视频-动作联合序列，DreamZero 则更进一步，用同一个 DiT 对视频和动作做联合扩散。两者都可以理解为"AR 处理历史上下文 + 扩散/FM 生成新内容"这个大范式在不同层次上的实例化。（见 **[WAM 专题](世界动作模型（WAM）.md) 第五节**）

## 七、Action Chunking：时序维度的动作预测

### 7.1 为什么需要

如果每次推理只预测下一步动作（单步策略），在 50Hz 控制频率下，模型需要每 20ms 完成一次完整的前向传播（视觉编码 + LLM + 动作解码），对大型模型来说延迟无法接受；而且独立的单步决策容易产生抖动——相邻帧的动作可能差异过大，机械臂运动不流畅。

### 7.2 做法

**Action Chunking**：每次推理一口气预测未来 $K$ 步的完整动作序列（称为"动作块"，action chunk），执行完这 $K$ 步后再发起下一次推理：

$$\text{一次推理} \rightarrow [a_t, a_{t+1}, \ldots, a_{t+K-1}] \quad \text{形状：} [K \times d_{\text{action}}]$$

推理频率降低 $K$ 倍，每步动作的计算代价不变，但控制更流畅（块内是联合优化的短期轨迹）。

### 7.3 Action Chunking 与动作表示的关系

Action Chunking（**时间维度**：预测几步）和动作的连续/离散表示（**空间维度**：每步动作如何编码）是两个正交的设计选择，互不依赖。实践中几乎所有方案都用**连续动作** + Chunking：

| 预测头类型 | 如何生成 [K, d] 输出 | 代表 |
|-----------|-------------------|------|
| **Transformer 解码器** | 自回归逐步生成 K 个动作 Token，每步输出 d 维向量 | ACT（Action Chunking with Transformers）|
| **Flow Matching / 扩散** | 对整个 [K, d] 矩阵整体去噪，一次性输出 | Diffusion Policy、π₀、LingBot-VA |
| **MLP** | 直接展平输出 K×d 维向量（忽略步间依赖） | 早期简单方案 |

Transformer 解码器能建模块内步间依赖，但自回归推理慢；Flow Matching 能建模多峰分布且支持整块并行生成，是目前最主流的方案。

### 7.4 代价：反应性缺口

Action Chunking 的核心代价是**反应性缺口（Reactivity Gap）**：执行期间如果环境变化（物体被意外碰歪、人突然出现），正在执行的动作块已经固定，无法中途修正。

典型的 $K$ 值权衡：
- **π₀**：$K=50$（约 1 秒动作块，流畅但反应慢）
- **LingBot-VA**：$K=4$（更短，快速校正，但推理更频繁）
- **ACT**：$K=100$（适合长时精细操作）

## 八、数据体系：机器人演示数据集

VLA 的数据生态是理解整个领域的关键背景。

### 8.1 为什么数据如此重要且稀缺

与互联网图文数据（规模：万亿 Token）相比，机器人演示数据极其稀缺：
- 收集成本高：每条演示需要人工操控机械臂完成任务，速度慢
- 异构性强：不同机器人平台的动作空间、摄像头配置、操作频率全不一样
- 覆盖有限：每个实验室只有 1–2 台机器人，场景和物体都很单一

这导致一个核心矛盾：**VLA 需要大规模多样性数据来训练泛化能力，但机器人演示数据天然稀缺且异构**。

### 8.2 Open X-Embodiment (OXE) 数据集

**OXE**（Google 联合多家机构，2023）是目前规模最大的跨机器人演示数据集：
- **规模**：~100 万条轨迹，来自 22 种不同机器人平台
- **多样性**：覆盖不同机械臂（Franka、UR5、Sawyer 等）、不同场景（厨房、仓库、桌面操作）
- **格式统一**：定义了跨机器人的标准动作空间（归一化的末端执行器位移 + 夹爪）

OXE 的关键贡献不只是规模，而是**证明了跨机器人联合训练能够提升泛化能力**——在 OXE 混合数据上训练的策略，在新机器人平台上的适配速度显著快于单平台训练。

### 8.3 其他重要数据集

| 数据集 | 机构 | 特点 |
|--------|------|------|
| **DROID** | Stanford + Berkeley 等 | 76k 条轨迹，13 个地点真实采集，高质量双摄像头 |
| **BridgeData V2** | Berkeley | 60k 条轨迹，家庭厨房场景，广泛用于基准测试 |
| **RoboMind** | 多机构联合 | 多语言任务描述，跨文化场景 |
| **LIBERO** | HuggingFace / 学术界 | 专为评估设计，4 个任务套件（空间/对象/目标/长时序） |

### 8.4 数据采集的工程挑战

**演示收集方式**：
- **遥操作（Teleoperation）**：人类戴 VR 手套或使用游戏手柄远程操控机械臂，同时记录摄像头图像和控制指令
- **示教（Kinesthetic Teaching）**：直接拖动机械臂示范轨迹，记录关节编码器数据
- **远程采集**：在多个地点部署相同设备，统一上传到中央数据库（DROID 的做法）

**跨平台的动作标准化**：不同机器人的动作空间不兼容（关节角度表示 vs. 末端执行器位移表示）。主流做法是统一转换为**末端执行器 delta 动作**（每步位移量 + 夹爪变化量），在语义层面对齐，允许模型学到跨平台通用的操作技能。

## 九、训练范式：预训练 + 后训练

### 9.1 两阶段训练

VLA 训练分为截然不同的两个阶段：

**第一阶段：预训练（Pretraining）**

目标是让模型获得跨任务跨平台的泛化能力。数据来源分两部分：
1. **视觉语言预训练**：VLM 骨干（视觉编码器 + LLM）用互联网图文数据（图文对、VQA 数据）预训练，学习视觉语义理解和指令跟随能力
2. **机器人运动预训练**：在 OXE 等大规模跨机器人数据集上训练动作解码器，学习通用的操作技能

关键工程决策：**是否联合训练（Co-training）**？

- **顺序训练**（先 VLM 预训练 → 再机器人数据微调）：简单，但 VLM 的语言理解能力可能在机器人微调中被"遗忘"（灾难性遗忘，Week 5 详细讨论）
- **联合训练（Co-training）**：VLM 数据和机器人数据按比例混合同步训练。可以缓解遗忘，但需要精心设计数据采样比例

π₀ 的实践表明：**大规模联合预训练（含 VLM 数据 + 机器人数据）是性能最优的方案**，但计算成本也最高。

**第二阶段：后训练（Post-training / Fine-tuning）**

用目标平台的少量演示做针对性适配：
- **数据量**：通常 50–500 条演示（相比预训练的数十万条极少）
- **训练设置**：更低的学习率（$10^{-5}$ vs. $10^{-4}$），更少的步数
- **目标**：在保留预训练泛化能力的同时，适配特定机器人平台、特定操作场景

类比：预训练是让模型"读了很多书、见过各种机器人任务"，后训练是"在目标工厂实习几天"。

### 9.2 参数高效微调（PEFT）在 VLA 中的应用

VLA 的 LLM 骨干通常有 7B–70B 参数，全量微调计算成本极高。实践中广泛使用 **LoRA**（Low-Rank Adaptation）：

- 冻结 LLM 骨干的全部权重
- 在注意力层的 Q、V 矩阵旁插入低秩增量矩阵（$\Delta W = BA$，$r = 8$–$64$）
- 只训练少量新增参数（约原模型的 0.1%–1%）

实践经验：
- **视觉编码器通常也部分冻结**——视觉特征在新平台上往往不需要大幅调整
- **动作解码器完全训练**——它是新添加的组件，权重需要从头学习
- LoRA rank 选择：$r=32$ 或 $r=64$ 在 VLA 微调中是常见配置

> **📖 关联讲义：Week 3 Part 1**，LoRA 的原理（低秩分解、$\alpha/r$ 缩放因子、为什么 B 初始化为零）和 SFT 实战细节在讲义中有完整推导。

### 9.3 多任务 vs. 单任务微调

一个实践细节：微调时是否需要**混入原始预训练数据**？

- **纯目标数据微调**：简单，但更容易发生灾难性遗忘，模型对预训练任务的能力退化
- **混合微调（replay-based）**：按比例（通常 1:9 到 1:4）混入部分原始预训练数据，显著缓解遗忘，但需要保留原始数据集的访问权

> **📖 关联讲义：Week 5 Part 1**，灾难性遗忘的机制和持续学习策略（EWC、Replay、Progressive Freezing）在 Week 5 中有系统讨论，VLA 的 Co-training 可以视为一种多任务持续学习场景。

## 十、代表性模型演进

VLA 领域的代表性工作并非孤立发表，而是形成了几条清晰的**演化脉络（series）**。理解这些脉络，比记住单个模型更有助于把握领域走向。

### 10.1 Google/DeepMind：RT 系列

Google 是 VLA 领域最早的系统性推动者，RT 系列奠定了多项基础范式。

**RT-1（2022）**：从零验证"机器人也能走大数据路线"。核心贡献不是架构，而是数据——在 700+ 任务、130,000+ 真实演示上训练，证明**任务多样性**是泛化的关键。动作解码使用 EfficientNet + Transformer 回归，方法相对朴素。

**RT-2（2023）**：战略转变的里程碑。不再从头训练视觉-语言骨干，而是直接把 PaLI-X（Google 55B VLM）迁移为机器人控制器，用 Token 离散化将连续动作变成语言 Token 输出。首次在机器人任务上展示了"VLM 中的世界知识"能直接迁移到操作决策（例如"易腐烂的东西放哪里？"——模型能从语言预训练中推断出冰箱）。代价是闭源、推理慢、动作精度受量化损失限制。

**Open X-Embodiment / RT-X（2023）**：与学术界合作建立**跨机器人数据集**（22 种机器人形态、527 个任务），验证了跨形态训练的正迁移：在混合数据上训练的策略在大多数机器人上优于单一数据训练。核心洞见是数据的**形态多样性**比数据量本身更重要。

**RT-H（2024）**：引入**层级动作语言（Action Hierarchy）**——LLM 先输出高层语言子目标（"移动到杯子旁边"），再由低层策略把子目标转为精确动作序列，解决了 RT-2 难以处理长时序任务的问题，但增加了训练标注成本（需要层级标签）。

```
RT 系列演化轴：
RT-1 ──→ RT-2 ──→ RT-X ──→ RT-H
（规模）  （VLM迁移）（跨形态）（层级规划）
```

### 10.2 Physical Intelligence：π 系列

Physical Intelligence（PI）是 2023 年由前 Google/DeepMind/Stanford 研究员创立的机器人 AI 公司，π 系列是目前**商业化最进取**的 VLA 产品线。

**π₀（pi-zero，2024）**：选择 Flow Matching 而非扩散策略（推理步数从 DDPM 的 100 步降至 10 步以内），在 Open X-Embodiment + PI 专有数据上做大规模预训练，再微调到具体硬件。核心卖点是**跨任务泛化**——同一套权重能部署到折叠衣服、装盘子、整理桌面等多种任务，成为 2024 年学术界公认的最强泛化基线（尽管闭源）。

**π₀.5（2025）**：显著加强**语言推理能力**。π₀ 接受短指令（"把杯子放到托盘"），π₀.5 能理解长指令和上下文（"桌上如果有脏盘子先收走再擦桌子"）。实现方式是在 LLM 骨干中引入更深的 Chain-of-Thought 中间推理，动作解码仍用 Flow Matching。代表了 VLA 从"执行者"向"能理解意图的助手"的升级。

```
π 系列演化轴：
π₀ ──────────────→ π₀.5
（泛化性+FM）        （长时序语言推理）
```

### 10.3 学术开源生态：Diffusion Policy / ACT / OpenVLA 脉络

这条脉络由多个学术团队平行推进，彼此技术借鉴，共同构成了 VLA 的**开源基础设施**。

**Diffusion Policy（2023，MIT/Columbia/UNC）**：从控制理论角度切入，把扩散模型引入机器人动作生成，系统验证了扩散解码在精细操作（插销钉、翻饼）上显著优于 MSE 回归。它本身不是完整的 VLA（通常用 ResNet 提取视觉特征，没有语言接口），但奠定了"扩散策略"范式，后续所有扩散/Flow Matching 解码器都直接继承其设计。

**ACT（Action Chunked Transformers，2023，Berkeley/Stanford + Aloha）**：在双臂机器人 Aloha 上提出两个核心贡献：Action Chunking（一次预测 k 步动作序列）和 CVAE 动作先验（建模多峰分布）。代码和 Aloha 硬件均完全开源，催生了大量双臂操作研究。

**Octo（2023，Berkeley）**：定位"机器人领域的 BERT"——在 Open X-Embodiment 全量数据上大规模预训练，Transformer 主干 + 扩散动作头，可直接用作各种机器人任务的微调起点。首个系统性验证"预训练-微调范式在机器人领域可行"的开源工作。

**OpenVLA（2024，Stanford + 多机构）**：将 7B 参数的 VLM（LLaMA-2 骨干 + SigLIP 视觉编码器）以 Token 离散化接口转为 VLA，在 Open X-Embodiment 上训练后**开源完整权重**。目标是提供"下载即可微调"的通用起点，现已成为学术界最常用的 VLA 基线。

**OpenVLA-OFT（2025，Stanford 等）**：在 OpenVLA 基础上把动作解码头换成 Flow Matching，同时引入并行解码加速推理。目前开源阵营中精度最高的基线，代表了开源社区对 Token 离散化的正式告别。

```
学术开源脉络：

Diffusion Policy ──────────────────────────────→ (影响所有扩散/FM解码器)
      │
      ↓
ACT（Chunking + CVAE）   Octo（预训练-微调范式）
                                    ↓
                          OpenVLA（7B 开源权重，Token 解码）
                                    ↓
                          OpenVLA-OFT（FM 解码头，当前开源 SOTA）
```

### 10.4 其他值得关注的工作（2024–2026）

**GR-1 / GR-2（字节跳动，2023–2024）**：将视频生成预训练与机器人策略学习统一。先用大量互联网视频训练视频预测模型，再迁移到机器人控制，让策略自然继承视频模型学到的物理常识。GR-2 进一步扩大预训练视频数据规模，在多个基准上优于同等规模的纯机器人数据方案，代表了"视频→物理常识"迁移路线的系统验证。

**Helix（Figure AI，2025）**：人形机器人公司 Figure 的 VLA 产品，专门针对人形机器人的**全身协调控制**（手臂 + 腿 + 腰部联动）。采用双层架构：高层 VLM 负责任务理解和规划，低层高频控制器负责全身运动执行。是 VLA 向全身人形控制延伸的代表性工作。

**Magma（Microsoft，2025）**：定位"统一多模态智能体基础模型"——同一套模型既能做 Web Agent（操作浏览器），也能做机器人操作。核心技术是**空间感知预训练（Spatial Awareness Pretraining）**，让模型在图像上建立坐标感，为 2D Web 操作和 3D 机器人操作提供统一的空间理解基础，代表了"机器人 VLA 与通用 Agent"合并的探索方向。

**LingBot-VA（蚂蚁集团，2026）**：将世界模型预训练与 VLA 在同一个网络中联合训练，用逆动力学模型从无标注视频中提取动作监督，直接针对"表示纠缠"（Representation Entanglement）问题（详见第十二节）。

### 10.5 横向对比

| 系列 / 工作 | 动作解码 | 数据策略 | 模型规模 | 开源 | 核心优势 | 主要局限 |
|------------|---------|---------|---------|------|---------|---------|
| RT-2 | Token 离散化 | VLM 微调 | 55B（PaLI-X） | ❌ | 世界知识直接迁移 | 精度受量化限制，推理慢 |
| RT-H | Token 离散化 | 层级微调 | RT-2 基础上 | ❌ | 长时序层级规划 | 需层级标注，成本高 |
| Diffusion Policy | 扩散策略 | 任务专属 | 小（ResNet） | ✅ 框架 | 精细操作精度高 | 无语言接口，推理慢 |
| ACT | CVAE + Chunking | 任务专属 | 小（ViT-B） | ✅ 框架+权重 | 双臂精细操作 | 泛化性弱，需专属数据 |
| Octo | 扩散策略 | OXE 预训练 | 93M | ✅ 权重 | 首个可微调预训练基线 | 较老，精度已被超越 |
| OpenVLA | Token 离散化 | OXE 预训练 | 7B | ✅ 权重 | 语言理解强，下载即用 | 动作精度中等（量化损失） |
| OpenVLA-OFT | Flow Matching | OXE 预训练 | 7B | ✅ 权重 | 开源精度最强 | 推理比 Token 解码慢 |

| GR-2 | 扩散策略 | 视频预训练 | 中等 | 部分 | 视频物理常识迁移 | 视频-动作域差距仍存 |

**几条横向洞见：**

1. **动作解码的迭代方向很清晰**：Token 离散化（RT-2 路线）因精度损失正在被 Flow Matching 替代，2025 年后几乎所有新工作都默认 FM 解码。

2. **开源与闭源的能力差距正在缩小**：2023 年闭源的 RT-2 是无可争议的 SOTA；2025 年开源的 OpenVLA-OFT 在多个基准上已能与 π₀ 竞争。

3. **规模不是唯一变量**：π₀（~3B 动作解码部分）在 55B 的 RT-2 上仍有显著优势，表明**数据多样性和解码方式**比纯粹的模型规模更重要。

4. **超越单步执行是共同趋势，但路线不同**：这批工作都在解决"VLA 只会当下一步"的问题，但方向分为两类。RT-H 和 π₀.5 走的是**语言规划路线**——用 LLM 的推理能力把长任务拆成语言子目标，属于"任务理解层"的扩展，不涉及物理动力学。GR-2 和 LingBot-VA 走的是**世界模型路线**——通过视频预测预训练或联合动力学建模，让模型真正"预测未来物理状态"。两类路线并非对立，但后者才是 12 节所讨论的根本性方向转换。

**开源可用性说明**（三种不同含义）：

| 类型 | 代表 | "开源"的实际含义 |
|------|------|---------------|
| **可下载权重直接微调** | OpenVLA、OpenVLA-OFT、Octo | 直接用作新机器人/任务的微调起点 |
| **开源框架，需自行训练** | Diffusion Policy、ACT | 照此方法从头训练，不是下载即用 |
| **闭源，仅论文数字** | RT-2、π₀/π₀.5、LingBot-VA | 只能在论文中引用其指标作对比参照 |

## 十一、评估体系

### 11.1 为什么 VLA 评估很难

机器人任务评估面临两个根本困难：
1. **真实机器人昂贵且不可复现**：不同实验室的机器人、物理环境、光照条件不同，真实成功率没有可比性
2. **仿真与真实存在差距**：纯仿真评估的成功率往往不能预测真实机器人上的表现

为此，社区逐渐形成了若干**标准化评估协议**：

### 11.2 主要评估基准

**LIBERO**（Learning Benchmark from Real Operationally）：

目前使用最广泛的仿真评估套件，分 4 个任务套件：
- **LIBERO-Spatial**：同一物体，不同空间位置（测试位置泛化）
- **LIBERO-Object**：同一动作，不同物体（测试物体泛化）
- **LIBERO-Goal**：相同物体，不同目标状态（测试语义泛化）
- **LIBERO-Long**：长时序多步骤任务（测试规划能力）

指标：每个任务的**任务成功率（Task Success Rate, TSR）**，通常运行 20–50 次取均值。

**SimplerEnv**：

专为评估"真实机器人预训练模型"设计，在 Google Scanned Objects（真实物体 3D 扫描）构建的高保真仿真环境中测试，与真实机器人评估的相关性（Spearman r ≈ 0.9）在目前所有仿真基准中最高。是评估 RT-2 类模型的主流选择。

**BridgeData 真实评估**：

用于 BridgeData 数据集对应的真实厨房环境。模型需要在真实 Widow-X 机械臂上执行任务，是目前最权威的"真实机器人"评估之一（代价是只有少数实验室能复现）。

### 11.3 评估局限

即使是最标准化的评估也存在局限：
- LIBERO 的仿真场景与真实物理差距明显（材质、质量、碰撞不真实）
- 成功率定义模糊（"放到托盘上"算成功的标准是什么？）
- 很多工作只报告部分套件，阻碍横向比较

## 十二、根本局限与世界模型路线

### 12.1 表面症状：三个执行层缺陷

即使是最好的 VLA（如 π₀.5），也面临一个**结构性困境**：同一个网络被迫同时完成视觉理解、物理动力学建模和运动控制三件性质不同的事。LingBot-VA 论文称之为**表示纠缠（Representation Entanglement）**，具体体现为三个执行层缺陷：

| 缺陷 | 含义 | 根本原因 |
|------|------|---------|
| **反应性缺口**（Reactivity Gap） | 执行动作块期间若环境变化，无法中途修正 | 动作在生成时已固定，不接实时反馈 |
| **长时记忆缺失**（Limited Long-term Memory） | 块与块之间"失忆"，难以完成需要跨块记忆的任务 | 每块独立生成，历史隐状态不持久 |
| **因果性违背**（Causality Violation） | 块内双向注意力允许"偷看未来"动作 | 物理世界中当前状态只依赖过去，不依赖未来 |

但这三个缺陷只是症状。要理解为什么 VLA 会有这些症状，需要深挖两个更根本的架构矛盾。

### 12.2 根本矛盾一：模仿学习的天花板

VLA 本质上是**行为克隆（Behavioral Cloning）**——它学习的是"见过这个场景时，专家怎么做"的条件概率 $p(\text{action} \mid \text{observation})$。这个框架有一个无法回避的理论上限：

**分布偏移（Distribution Shift）导致的错误级联。** 训练数据全部来自专家演示，专家几乎不犯错，因此训练集中几乎没有"从错误状态恢复"的样本。真实执行时，只要出现任何微小偏差（抓手角度差 2°，物体比预期重），模型就进入了**从未见过的状态**，后续每一步的误差只会越来越大，直到任务彻底失败。

这不是数据量的问题，而是行为克隆的**结构性问题**：你只学会了"理想情况下怎么做"，没学会"出错后怎么恢复"。扩大数据规模可以推迟这个问题，但无法消除它。

相比之下，如果模型学会了"这个动作会导致什么结果"（即物理世界的**动力学**：与"静力学"相对，描述世界状态如何随时间和动作演化，形式上是 $s_{t+1} = f(s_t, a_t)$——给定当前状态和执行的动作，下一时刻变成什么），它就可以在执行过程中持续预测、观察、纠偏——这正是世界模型的核心能力。

### 12.3 根本矛盾二：两个目标占用同一表示空间

VLA 的 LLM 骨干是从语言预训练继承的，它的表示空间被优化用来理解语义（"把杯子"="移动一个容器"）。但精确的运动控制需要的是另一种表示——精确的空间坐标、关节角速度的物理量，和语义理解是完全不同的信息粒度。

当 VLA 同时用这一套参数做"理解任务含义"和"输出精确关节角度"时，两个目标的梯度会**相互竞争**，网络被迫在"能说清楚做什么"和"能精确算出怎么做"之间折中，两头都不彻底。

这个矛盾随着任务精度要求的提高会越来越突出——插 USB 插口（1mm 精度）和"拿起杯子"（厘米级精度）对表示空间的需求完全不同，但 VLA 用同一套隐状态处理两者。

### 12.4 数据层的结构性劣势

除了架构矛盾，VLA 还面临一个数据层的根本劣势：**它能学习的所有东西都必须来自有动作标签的机器人演示数据**。

收集一条有动作标签的演示需要人工遥操作（Teleoperation）——每条数据都要真人实时控制机械臂，成本极高。Google、PI 等公司花费数年时间才积累了几十万条演示。而互联网上有数十亿条视频，其中包含海量的物理交互、物体运动、人手操作的信息——但这些视频**没有动作标签**，VLA 无法直接利用。

这意味着 VLA 的规模上限被物理世界中的数据采集速度卡住了，而这个速度远慢于互联网文本/视频的积累速度。

### 12.5 感知模态的缺失：触觉与力控

上述三个问题都可以被纳入 WAM 的解题框架，但还有一个结构性局限和 WAM 正交、独立存在：**VLA 只有"看"，没有"摸"和"感受力"。**

**当前 VLA 的感知几乎清一色是纯视觉。** 机械臂硬件上通常配有腕部六轴力矩传感器（如 Franka 自带），但 Open X-Embodiment、BridgeData 等主流训练数据集在采集时没有同步录制这些信号，导致模型完全不知道怎么利用它们。触觉传感器（GelSight、DIGIT 等）则更加小众，型号各异，没有形成跨机器人的统一标准。

**这一缺失在以下任务中会直接失效：**

| 场景 | 为什么视觉不够 |
|------|-------------|
| 抓取易碎物（鸡蛋、薯片袋） | 无法从图像判断当前夹紧力是否过大 |
| 插拔任务（USB、螺丝拧紧） | 毫米级对齐误差 + 接触力缺一不可，视觉分辨率不足 |
| 操作柔性物体（布料、面团） | 变形状态需要触觉感知，视觉难以捕捉接触面内部受力 |
| 手被遮挡时的盲抓 | 摄像头看不到，完全依赖触觉 |

**力控问题不只在感知端，也在动作端。** 当前 VLA 输出的是**位置/速度目标**，底层控制器跟踪这个目标。真正的力控需要输出**力目标或阻抗参数**（刚度、阻尼），是完全不同的控制模式，现有 VLA 的动作空间定义里几乎没有这一维度。

**当前探索方向：**

- 把触觉传感器（如 GelSight）的输出当作"另一路摄像头"输入视觉编码器——工程最简单，但丢失了力的绝对量
- 将力矩读数编码为数值 Token，与视觉 Token 拼接后送入 LLM
- 在动作输出端增加阻抗参数维度（目标刚度/阻尼），与位置目标一起输出
- DexVLA（字节，2025）在灵巧手场景下开始系统性地引入力反馈

**与 WAM 方向的关系**：触觉/力控和世界模型是**正交的两个维度**——WAM 解决的是"能否预测动作后果"，触觉/力控解决的是"能否感知接触时的物理状态"。完整的高精度操作系统两者都需要，但由于传感器标准化程度低、数据采集困难，触觉这条线当前主要在灵巧手（dexterous manipulation）和接触丰富操作（contact-rich manipulation）子领域独立推进，尚未融入主流 VLA 训练管线。

### 12.6 为什么 WAM 是结构性的解法

上述三个问题——分布偏移、表示纠缠、数据稀缺——在 WAM（World-Action Model）框架中都有对应的解法，且这些解法来自同一个核心思路的转变：

> **从"模仿专家动作"转向"理解世界动力学，再推导动作"。**

| VLA 的问题 | WAM 的解法 | 为什么有效 |
|-----------|-----------|-----------|
| 分布偏移，执行出错后无法恢复 | 学习正向动力学 $p(s_{t+1} \mid s_t, a_t)$，推理时持续预测-观察-纠偏 | 世界动力学是通用的，不依赖专家演示的状态分布 |
| 表示纠缠，语义理解与精控共用参数 | 世界模型负责语义+物理预测，动作头只负责从预测结果推导动作 | 两个目标分开优化，梯度不竞争 |
| 无标注视频无法利用 | IDM（逆动力学模型）从视频帧间差异反推动作，为无标注视频提取伪标签 | 训练数据从"机器人演示"扩展到"任意物理交互视频" |
| 开环执行，Action Chunk 期间无反馈 | 世界模型在每步生成虚拟"未来帧"，策略基于虚拟预测做闭环修正 | 即使物理执行中断，模型在"脑中"仍持续运行 |

值得注意的是，WAM 不是"VLA + 一个世界模型模块"的拼接，而是从训练目标层面的重构——**预测未来状态**和**生成控制动作**被统一在同一个联合损失里，而不是把视觉理解、动力学建模、动作生成三者硬塞进同一个 next-token-prediction 目标。

### 12.7 研究社区转向的信号

2024–2025 年以来，以下几个方向的汇聚表明 WAM 路线正在成为新的研究主轴：

- **视频生成模型的成熟**（Sora、CogVideoX 等）证明了大规模视频预训练模型能捕获物理常识，提供了现成的世界模型骨干
- **DreamerV3 / DIAMOND 等工作**验证了"用世界模型的想象空间训练策略"在 Atari 等环境有效，向机器人领域的迁移成为自然的下一步
- **IDM 技术的成熟**（LingBot-VA、VideoWorld 等实际使用）打通了"无标注视频→动作标签"的通道，数据层瓶颈开始松动
- **字节 GR-2、LingBot-VA 等工作**在实际机器人上展示了视频预训练→操作迁移的可行性，不再是纯理论设想

## 十三、前沿发展（2025–2026）

### 13.1 端到端基础模型方向

**Gemini Robotics**（Google DeepMind，2025）：基于 Gemini 2.0 的多模态能力，直接扩展到机器人控制。关键洞察：**共享视觉-语言预训练带来的常识推理能力**对机器人的意外处理（unexpected situations）至关重要。用 Gemini 骨干支持机器人的自然语言交互与视觉推理，在新物体新场景的零样本泛化上取得显著进展。

**Helix**（Figure AI，2025）：人形机器人专用 VLA，针对 50+ DoF 的高维动作空间设计。核心贡献是**层级动作解耦**：高层用 LLM 做任务规划（输出子目标），低层用独立的运动控制器执行精确动作，解决了单一 VLA 在高 DoF 机器人上精度不足的问题。

### 13.2 扩展定律应用

**π₀.5**（Physical Intelligence，2025）：系统验证了 VLA 领域的**扩展定律**——在 OXE 上的预训练规模越大、模型参数越多，在新任务上的零样本/少样本适配能力越强，定性趋势与 LLM 的 Chinchilla 定律类似（但机器人领域数据规模限制更严）。

### 13.3 数据飞轮：合成数据与世界模型仿真

最受关注的前沿方向之一：**用世界模型生成大量合成机器人演示数据**，打破真实数据稀缺的瓶颈。

- **IRASim**（字节跳动 + 北京大学）：视频扩散模型生成机械臂操作视频，用生成视频微调 VLA
- **UniSim**（Google）：可交互视觉仿真器，给定初始帧 + 动作序列，生成未来视频帧作为虚拟训练环境
- **Genesis**（多机构，2025）：物理仿真引擎 + 生成模型结合，追求"物理正确"的合成数据

目前最大挑战是**感知漂移（Perceptual Drift）**——合成数据视觉上逼真，但物理细节（摩擦力、惯性、接触动力学）不准确，策略在真实场景中失效。如何缩小这一差距是 2025–2026 年的核心工程问题。

### 13.4 推理时扩展（Test-time Scaling）

受 LLM 推理扩展定律（DeepSeek-R1、OpenAI o1）启发，探索在机器人任务中也做推理时扩展：

- **树搜索 + 世界模型**：用世界模型在推理时生成多条候选轨迹，奖励模型选出最优执行路径
- **Chain-of-Thought 动作规划**：让 VLA 的 LLM 骨干先输出文字规划（"我要先确认杯子位置，再调整抓握角度..."），再输出动作序列

> **📖 关联讲义：Week 7 Part 2**，推理时搜索（MCTS、Beam Search + 过程奖励）的通用框架在 Week 7 中有系统介绍，机器人规划可以视为该框架在具身域的应用。

### 13.5 开放式操作挑战

**Open-world manipulation**：现有 VLA 在分布内的任务表现接近人类，但遇到**从未见过的物体组合或指令**时仍显著下降。2025–2026 年的主要探索方向是：
- **更强的 LLM 骨干**（70B+ 参数）带来更好的组合泛化能力
- **持续学习**（Continual Learning）在部署后不断从新演示中更新，而不遗忘旧技能

### A.1 逆动力学模型（IDM）

> 关联正文：**第十二节 → 世界模型路线**

在世界模型和 WAM 的讨论中经常出现"逆动力学模型"（Inverse Dynamics Model, IDM）的概念，这里做简单介绍。

**正向动力学（Forward Dynamics）**：给定当前状态 $s_t$ 和动作 $a_t$，预测下一状态 $s_{t+1}$：
$$s_{t+1} = f_{\text{fwd}}(s_t, a_t)$$

**逆动力学（Inverse Dynamics）**：给定当前状态 $s_t$ 和下一状态 $s_{t+1}$，反推执行了什么动作 $a_t$：
$$a_t = f_{\text{inv}}(s_t, s_{t+1})$$

IDM 在 VLA 中的应用价值：大量**无标注视频**（不含动作标签）可以用来训练 IDM——模型观察两帧之间的视觉变化，学会推断"发生了什么动作"。这样就能从无标注互联网视频中提取隐式的动作监督信号，极大扩展训练数据来源。LingBot-VA 和 VideoWorld 都使用了这一思路。

### A.2 连续动作量化方案

> 关联正文：**第六节 → 直接回归路线**

RT-2 类模型将连续动作量化为离散 Token，常见方案：

**Uniform Binning（均匀分桶）**：将每个动作维度的值域均匀分成 256 个 Bin，取中心值作为该 Bin 代表的动作值。简单但对分布尾部精度低（机器人大部分动作集中在小范围，均匀分桶浪费了大量 Bin 在极少发生的区域）。

**Quantile Binning（分位数分桶）**：用训练集中该维度动作值的分位数划分 Bin，使每个 Bin 包含相同数量的训练样本。精度更均匀，目前是 RT-2 类模型的标准做法。

### A.3 多摄像头 × 多帧输入的编码设计

> 关联正文：**第四节 → 视觉编码器 → VLA 的特殊需求**

#### 多摄像头：用一个编码器还是多个？

**主流做法：权重共享，逐路独立编码，再拼接。**

为每路摄像头训练一个专属 ViT 既昂贵又难以对齐，实际方案是用**同一个 ViT 分别编码每路图像**，再把各路的视觉 Token 序列拼接后一起送入 LLM：

```
腕部摄像头图像 → ViT (shared) → [视觉 Token × 256]
全局摄像头图像 → ViT (same)  → [视觉 Token × 256]
                                         ↓
                            concat → [512 个 Token 进 LLM]
```

纯粹拼接时模型无法区分哪些 Token 来自哪路摄像头，需要额外的标识机制：

- **Camera-ID Embedding**：类似 BERT 的 segment embedding，给每路摄像头的所有 Token 加一个可学习的偏置向量：
  $$\text{token}_{\text{final}} = \text{patch\_embed} + \text{pos\_embed} + \text{camera\_embed}[i]$$
- **View Token**（更现代）：在每路视觉 Token 序列前插入一个可学习的 `[VIEW_i]` token，让模型用这个 token 汇聚该摄像头的全局语义，再参与跨摄像头交互。

#### Token 膨胀：多摄像头 × 多帧是核心工程挑战

这是多模态机器人任务最直接的工程压力：

$$3 \text{ 路摄像头} \times 5 \text{ 历史帧} \times 256 \text{ patch} = 3840 \text{ 视觉 Token}$$

加上语言指令的约 100 个 Token，总序列接近 4000。这直接拖垮推理速度，使模型无法满足控制频率要求（通常需要 ≥10 Hz）。主要压缩策略如下：

| 方法 | 原理 | 代价 |
|------|------|------|
| **Temporal Patch Pooling** | 对同一空间位置的 patch 跨时间做均值/最大池化，N 帧压成 1 帧 | 丢失帧间精细差异 |
| **Perceiver Resampler** | 用少量 Learnable Query 通过 Cross-Attention 从全部视觉 Token 中蒸馏信息，输出固定长度序列 | 引入固定容量瓶颈，但压缩率可控 |
| **Spatial Subsampling** | ViT 输出后对相邻 patch 做池化（如 2×2 → 1），空间分辨率减半 | 丢失细粒度空间信息 |
| **历史帧稀疏采样** | 只保留关键帧（每隔 N 帧采 1 帧）或仅保留 1–2 帧历史 | 时序上下文粗糙 |

实践中往往**组合使用**：先对历史帧做稀疏采样，再对每帧的空间 Token 做 Subsampling，使总 Token 数控制在 256–512 以内。

#### 当前帧 vs 历史帧：两者"角色"本质不同

历史帧不是当前帧的简单重复，它们承载的语义信息性质不同：

| 角色 | 承载信息 | 典型用途 |
|------|---------|---------|
| **当前帧** | 物体当前位置、姿态、遮挡情况 | 直接决策依据 |
| **短期历史（1–3 帧）** | 物体运动速度和方向、夹爪运动趋势 | 预测近期位置、避免控制震荡 |
| **中期历史（4–10 帧）** | 子任务完成度（夹爪是否已接近目标） | 任务进展判断、长程纠错 |

**如何在编码时体现这种角色差异？**

1. **Temporal Position Embedding**：给每一帧加上时间步嵌入，$t=0$（当前帧）、$t=-1$、$t=-2$……模型通过学习发现"$t=0$ 的 Token 应获得更高权重"。
2. **顺序即信息**：将历史帧按时间顺序排在序列前部，当前帧排在最后。在因果注意力掩码下，越靠近序列末尾的 Token 越能影响最终输出，当前帧天然获得最高的决策权重，无需额外设计。
3. **Temporal-Spatial 注意力解耦**（Video Transformer 风格）：ViT 内部先对每帧独立做 Spatial Attention，再跨帧做 Temporal Attention，两阶段解耦。但推理开销大，在需要实时控制的 VLA 中较少采用。

#### 时序对齐：不同摄像头的帧是否同步？

分两层考虑：

**硬件层**：工业机器人通常使用硬件触发同步——所有摄像头共享一个时钟信号，在同一时刻曝光。此时对齐由数据采集保证，模型无需处理。

**软件层**（异步摄像头场景）：若各路摄像头帧率不一致（如全局摄像头 30 Hz、腕部摄像头 60 Hz），需在送入模型前做软件对齐：取最近帧匹配（Nearest-Frame Matching）或线性插值到同一时间戳。这是**部署阶段的工程问题**，在模型训练数据中通常已经处理好，不体现在架构设计里。

#### 视觉控制频率与推理速度的矛盾

视觉编码器处理一帧需要数十毫秒，而机器人控制器要求 20–50 Hz 的高频动作输出。常见解法是**异步架构**：

- **视觉编码**低频运行（5–10 Hz），把当前帧 + 历史帧压缩为视觉特征向量，异步更新。
- **动作解码器**高频运行，从最新的视觉特征向量中连续预测动作，或使用 **Action Chunking**——一次预测未来 N 步的动作序列，中间无需视觉更新，直接按序执行。

Action Chunking 是 ACT（Action Chunked Transformers）提出的核心思路，现已成为高频控制场景的标准配置。

#### 多摄像头是否需要显式 3D 几何关联？

多路摄像头本质上是对同一 3D 世界的不同投影，理想情况下模型应能通过双目/多目三角定位推断物体的 3D 坐标。但大多数 VLA 并没有显式建立摄像头之间的几何关系，而是依赖大量数据让 LLM **隐式学习**视角-空间映射。

这是已知的弱点：遇到摄像头位置与训练时不同的场景，模型的空间感知会显著退化。部分工作（如 SpatialVLA，2025）开始引入**相机外参编码**（Camera Extrinsic Embedding）——把摄像头的旋转矩阵和平移向量编码为额外的位置嵌入附加给视觉 Token，让模型在坐标系层面理解"腕部摄像头看到的物体"和"全局摄像头看到的同一物体"之间的几何对应关系。

**VQ-VAE 码本**：用向量量化自编码器学习一个离散码本，用码本向量表示动作。相比分桶方法，码本能捕获动作维度间的相关性（不同维度的联合量化），但训练更复杂。
