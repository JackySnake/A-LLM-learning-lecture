# 视觉 Token 压缩：多模态大模型的"带宽"工程

*触发来源：学习 Week 8 Part 2.2（Qwen-VL 的 2×2 Merging）时，发现"压缩步骤"远不止 2×2 合并一种做法，而是一个独立、且 2024–2026 年极其活跃的研究方向。值得脱离单一的 Qwen 工程视角，用一个统一的分类学框架系统展开（2026-06）*

> **前置知识**：Week 8 Part 2（动态分辨率、2×2 Merging、多模态 Tokenizer）、附录 A.9（视觉编码器构建）、Week 6（KV Cache 与推理优化）。本文把 Week 8 一笔带过的"在 ViT 和 LLM 之间必须有一个压缩步骤"打开成一个完整的设计空间。

---

## 一、为什么必须压缩：先算一笔 Token 经济账

视觉 Token 压缩不是"锦上添花的优化"，而是高分辨率多模态模型**能不能跑起来**的前提。先感受数量级（ViT patch=14）：

| 输入 | 原始视觉 Token 数 | 占上下文 |
| :--- | :--- | :--- |
| 224×224 缩略图 | 256 | 还好 |
| 1080P 图像 | ~10,500 | 吃掉一个万级窗口 |
| 4K 图像 | ~42,000 | 单图就爆上下文 |
| 1 分钟视频（32 帧 720P） | ~53,000 | 完全不可接受 |

为什么这是灾难？因为代价不是线性的，而是**踩在三条非线性曲线上**：

1. **Attention 是 $O(n^2)$**：Token 翻 10 倍，注意力计算翻 100 倍。
2. **KV Cache 是 $O(n)$ 显存 + $O(n)$ I/O**：长视频的 KV Cache 能轻松吃掉几十 GB（见 Week 6）。
3. **Prefill 延迟是 $O(n^2)$**：视觉 Token 全在 prompt 里，首 Token 延迟（TTFT）被视觉 Token 数死死拖住。

> **一句话定性**：视觉 Token 压缩，本质是在"**看得清**"（高分辨率、多 Token）和"**算得动**"（低延迟、省显存）之间找平衡点。它是多模态系统的"带宽工程"——决定了一张高清图、一段长视频能不能进得了模型、跑得起推理。

而且有一个**先验事实**让压缩天然可行：**图像的信息密度极不均匀**。一张 PPT 截图，大片白背景的 Token 携带的信息趋近于零，而一行小字的 Token 信息密度极高。视觉 Token 里存在大量冗余——压缩要做的，就是把冗余挤掉、把信息留下。

## 二、一张地图：压缩可以发生在四个位置

理解整个领域的钥匙，是先看清**压缩可以插在流水线的哪个环节**。把多模态 pipeline 摊开：

```text
像素图像
   │
   ▼
┌─────────┐   ① ViT 编码器内部压缩
│  ViT    │ ◄─── （token 还在视觉空间时就合并）
└─────────┘
   │  视觉特征序列
   ▼
┌─────────┐   ② Projector / Connector 压缩  ★最主流
│Projector│ ◄─── （进 LLM 之前，把 N 个压成 M 个）
└─────────┘
   │  对齐后的视觉 Token
   ▼
┌─────────┐   ③ LLM 解码器内部剪枝
│  LLM    │ ◄─── （在 decoder 的某些层之后丢弃视觉 token）
│ decoder │
└─────────┘   ④ KV Cache 压缩
   │       ◄─── （推理时压缩已缓存的视觉 KV）
   ▼
  输出
```

| 位置 | 代表方法 | 需要训练？ | 特点 |
| :--- | :--- | :--- | :--- |
| ① ViT 内部 | ToMe（Token Merging） | 免训练 | 通用、但对 VLM 任务感知弱 |
| ② Projector ★ | 2×2 MLP / Pixel Shuffle、Q-Former、Perceiver、TokenPacker、C-Abstractor | **多数需训练** | 主流落地方案，压缩是模型结构的一部分 |
| ③ LLM 内部 | FastV、PyramidDrop、SparseVLM、VisionZip | **多数免训练** | 即插即用，对已有模型零成本提速 |
| ④ KV Cache | ZipVL、ElasticCache | 免训练 | 与 LLM 通用 KV 压缩同源 |

**这张表是整个专题的骨架。** 记住一个直觉对偶：**位置②（Projector）的方案大多"需要训练、一次成型"，位置③（LLM 内部）的方案大多"免训练、即插即用"**——这决定了它们各自的适用场景，后面会反复呼应。

## 三、两大范式：合并（Merge）vs 剪枝（Prune）

无论压缩发生在哪个位置，动作只有两类。这是另一根理解主轴：

**合并 Merging：$N \to M$，信息融合，不丢弃。**
把多个 Token 用某种方式（拼接 / 加权 / cross-attention）融成一个。Qwen 的 2×2 MLP Merge、ToMe、Q-Former 都属此类。
- 优点：信息不"硬丢"，全局内容保留得好。
- 缺点：融合可能"糊"——一行小字和旁边的空白被合进一个 Token，细节被稀释（这正是 Week 8 讲"为什么不用 Average Pooling"背后的同一个隐忧）。

**剪枝 Pruning：$N \to M$，直接丢弃"不重要"的 Token。**
评估每个 Token 的重要性，扔掉低分的。FastV、PyramidDrop、FasterVLM 属此类。
- 优点：留下来的 Token 保持原始"锐利度"，不被稀释。
- 缺点：丢错了就彻底没了——如果被剪的 Token 恰好含关键信息（那行小字），无法挽回。

> **合并 vs 剪枝的本质张力**：合并是"有损压缩里的平均主义"（人人都留一点，但都模糊了），剪枝是"精英主义"（少数留全、多数清零）。近年很多 SOTA（如 VisionZip）干脆**两者结合**：少数高价值 Token 完整保留（剪枝思路），剩下的大量低价值 Token 合并成几个"上下文摘要"（合并思路）。

### 怎么判断一个视觉 Token "重不重要"？

这是所有压缩方法的核心命题，业界用四类信号：

| 信号 | 含义 | 代表方法 |
| :--- | :--- | :--- |
| **[CLS] 注意力** | ViT 的 CLS Token 对各 patch 的注意力 = 该 patch 对全局语义的贡献 | FasterVLM、HiRED |
| **文本→视觉交叉注意力** | LLM 里文本 Token 对各视觉 Token 的注意力 = 该视觉 Token 对当前问题的相关性 | FastV、SparseVLM |
| **视觉特征冗余度** | Token 之间的相似度，相似的就是冗余 | ToMe、VisionZip |
| **指令相关性** | 由用户问题决定保留哪些区域（问"左下角写了啥"就保左下角） | 指令引导类（见第七节） |

## 四、位置②详解：Projector 压缩（主流、需训练、一次成型）

这是工业界**实际落地最多**的位置——压缩直接长在模型结构里，训练好就固定下来。三大流派：

### 4.1 Pooling 派：保留空间结构

核心思想：把**空间相邻**的 Token 聚合。因为图像的相邻区域本就高度相关，按空间分组合并最自然。

- **2×2 MLP Merge（Qwen-VL）/ Pixel Shuffle（InternVL）**：2×2 相邻块 Concat 成 $4d$，再 MLP 投影回 $d$。固定 75% 压缩。两家独立设计、数学等价（Pixel Shuffle 源自超分辨率的 ESPCN）。详见 Week 8 Part 2.2。
- **C-Abstractor（Honeybee, CVPR 2024）**：用 **ResNet 风格的卷积块**做下采样。动机：MLP/Q-Former 会破坏图像的**局部性（locality）**，而卷积天然保局部——这对 OCR、grounding 这类空间敏感任务很关键。
- **LDP / LDPv2（MobileVLM）**：轻量级深度可分离卷积下采样，主打端侧效率。

> Pooling 派的共同信仰：**空间结构是视觉的命根子**，压缩绝不能打乱"谁在左上、谁在右下"。

### 4.2 Query 派：内容选择，丢掉空间网格

核心思想：用一组**可学习的 Query**，通过 Cross-Attention 从视觉特征里"主动抽取"信息，输出**固定数量**的 Token。

- **Q-Former（BLIP-2）**：32 个 learnable query，把任意多的视觉 Token 压成固定 32 个。压缩比极高（~97%）。
- **Perceiver Resampler（Flamingo）**：N 个 latent query 做 cross-attention 重采样，能优雅处理变分辨率。

**Pooling 派 vs Query 派的根本分歧**：

| | Pooling 派 | Query 派 |
| :--- | :--- | :--- |
| 保留什么 | **空间结构**（网格还在） | **语义内容**（网格被打散） |
| 压缩比 | 固定（如 75%） | 可极高（→固定 32 个） |
| 短板 | 压缩比不灵活 | 信息瓶颈严重，高分辨率细节/OCR 损失大 |
| 现状 | **当前主流**（Qwen/InternVL） | 热度下降（Q-Former 逐渐被弃用） |

> 这条分歧线呼应 Week 8 附录 A.6（Projector 演进）：行业从复杂的 Q-Former/Resampler **回归到了简单的 MLP+Pooling**——因为实践发现，对于高分辨率理解，"保住空间结构"比"激进压缩"更重要。

### 4.3 Coarse-to-Fine：TokenPacker

**TokenPacker（2024）** 是一个精巧的折中：先把视觉特征插值成**低分辨率的 point query**（粗粒度全局视图），再用 **region-to-point injection** 把高分辨率的局部细节"注入"回这些 query。相当于"先看整体、再补细节"，在高压缩比下仍较好保留细粒度信息。

## 五、位置③详解：LLM 内部剪枝（免训练、即插即用）

这是 2024 年后**论文数量爆炸**的方向，因为它有一个无法抗拒的卖点：**不用重新训练，对任何现成 VLM 直接加速**。

### 5.1 开山洞察（FastV）：LLM 深层根本"不看"视觉 Token

**FastV（ECCV 2024）** 的核心发现极其反直觉也极其重要：

> 在 LLM 的浅层（前 2 层左右），文本 Token 会大量注意视觉 Token；但**从第 2~3 层之后，文本对视觉 Token 的注意力急剧衰减到接近 0**。也就是说，视觉信息在浅层就已经被"吸收"进文本 Token 的表示里了，深层继续携带成百上千个视觉 Token，纯属**计算浪费**。

于是 FastV 的做法简单粗暴：**在第 K 层（如第 2 层）之后，按注意力分数剪掉大部分视觉 Token**，只留少数。training-free，却能砍掉一半以上 FLOPs 而精度几乎无损。这个洞察奠定了整条 training-free 剪枝路线。

### 5.2 后续演进

- **PyramidDrop（CVPR 2025）**：FastV 是"一刀切某层全剪"，PyramidDrop 改为**分层递进式**——浅层多留、越深越激进地 drop，像金字塔一样逐层收窄。更贴合"信息逐层被吸收"的规律。
- **SparseVLM / FastVLM 等**：在重要性度量、剪枝调度上做改进。
- **FasterVLM / HiRED**：把判断提前到**进 LLM 之前**，用 ViT 的 **[CLS] 注意力**给 Token 排序，早早剪掉。
- **VisionZip（CVPR 2025）**：进 LLM 前，按特征相似度把 Token 分成**少数 dominant Token（完整保留）+ 大量 contextual Token（合并成几个摘要）**——前述"剪枝+合并结合"的代表。

> **为什么这派天生免训练？** 因为它不改变模型的任何权重，只是在推理时"少算一些 Token"。模型本来就见过完整 Token，少喂几个不会让它"不认识"——前提是别把关键 Token 剪掉。这也正是它的阿喀琉斯之踵（见第八节）。

## 六、位置④：KV Cache 压缩（推理时的最后一道关）

视觉 Token 即使进了 LLM，它们的 Key/Value 还会**常驻 KV Cache**，在后续每一步解码中被反复读取。位置④就是压缩这部分缓存。

- **ZipVL**：根据**注意力分布动态决定**保留多少视觉 KV，并对保留的部分做量化。它同时跨越了"剪枝"和"KV 压缩"两个范畴。
- 与 Week 6 学过的通用 KV Cache 压缩（H2O、SnapKV 等）同源——只不过这里专门针对"视觉 KV 比文本 KV 更冗余"这一特性做优化。

> 直觉：位置③省的是**算力（FLOPs）**，位置④省的是**显存与访存带宽（KV I/O）**。长视频解码场景下，访存往往比算力更早成为瓶颈，所以④越来越重要。

## 七、自适应压缩：让模型"看内容定压缩比"

前面的方案（尤其 Pooling 派）大多用**固定压缩比**——对白背景和密集表格一视同仁。这显然是浪费。2025–2026 最活跃的子方向就是**自适应/动态压缩**：

- **ZipVL**：逐层依注意力分数分布**动态**决定该层压缩率。
- **ATP-LLaVA**：在 LLM 解码器层内部插入**自适应剪枝模块**，每层、每样本自己决定剪多少。
- **EvoComp**：用语义引导的进化式标注，训练一个**轻量压缩器**。
- **指令引导（Instruction-guided）压缩**：最有前景的思路之一——**根据用户的问题**决定保留哪些 Token。问"图里的车牌号是多少"，就重点保留车牌区域的 Token，其余狠狠压。把"压缩"从一个固定的预处理，变成一个**与任务耦合的、动态的注意力分配问题**。

> 自适应压缩的哲学转变：从"**先验地、均匀地**压"（2×2 一刀切），转向"**后验地、按需地**压"（看内容、看问题）。代价是训练/工程复杂度上升，这也是它尚未完全取代固定方案成为工业默认的原因。Week 8 Part 2.2 末尾提到的 2026 TMLR 综述与 CVPR 2026 多篇论文，主体都在这个方向上。

## 八、关键张力与评测陷阱（最容易踩的坑）

这一节是"避坑指南"，也是面试时能体现深度的地方。

**① 任务敏感度天差地别。** 同一个压缩方法，在不同任务上的损失可能差一个数量级：

| 任务类型 | 对压缩的敏感度 | 原因 |
| :--- | :--- | :--- |
| 自然图像 caption / MMBench | **低**（压 75% 几乎无损） | 全局语义即可，不需要每个像素 |
| VQA（一般问答） | 中 | 看问题，部分需要细节 |
| **DocVQA / OCR / 表格 / UI** | **极高**（激进压缩直接崩） | 细粒度文字，丢一个 Token 可能丢一个字 |
| **Grounding / 检测** | **极高** | 需要精确空间定位 |

**② 评测陷阱：警惕"无损压缩"的宣传。** 很多论文只在**不敏感的 benchmark**（MMBench、自然图像 VQA）上报告"压缩 90% 性能无损"，但一换到 **DocVQA / TextVQA** 就大幅掉点。**看到"无损压缩"先问一句：在什么任务上无损？** 一个压缩方案的真实价值，要看它在**敏感任务**上的压缩比-性能曲线拐点在哪。

**③ 视频是另一个量级的机会。** 视频的冗余不在帧内，而在**帧间**——相邻帧几乎一模一样。所以视频压缩的重点是**时序去冗余**（如 Qwen 的 3D 时序 Pooling、相邻帧合并），潜在压缩比远高于单图。这也是长视频理解能跑起来的关键。

## 九、选型决策表

| 你的处境 | 推荐路线 |
| :--- | :--- |
| 从头训 VLM，要稳妥的主流方案 | 位置②的 **2×2 MLP Merge / Pixel Shuffle**（保空间结构，工业验证充分） |
| OCR / 文档 / grounding 为主 | 低压缩比 + 保局部性（**C-Abstractor**），慎用 Query 派；或干脆走原生高分辨率少压缩 |
| 已有现成 VLM，想免训练提速 | 位置③的 **FastV / VisionZip**，即插即用 |
| 长视频推理，显存/延迟吃紧 | 时序去冗余 + 位置④ **KV Cache 压缩（ZipVL）** |
| 追求极致效率、能承担工程复杂度 | 位置③/⑦ 的**自适应 / 指令引导**压缩 |

**一条总原则**：**压缩比和任务敏感度要匹配**。自然图像可以激进压；文档 OCR 必须保守。没有"万能压缩比"，只有"任务相关的最优压缩比"。

## 十、前沿与开放问题（2026）

1. **Training-free 会不会被"原生稀疏"取代？** 当前 training-free 剪枝是"事后补救"。更彻底的思路是让模型在**预训练时就学会用更少的视觉 Token**（原生稀疏视觉表征），从根上避免冗余。
2. **与生成式的关系。** 本专题讲的是"理解侧"的压缩。但 Transfusion（Week 8 附录 A.7）这类**连续生成**模型，图像 Token 既要"看"也要"生成"——压缩会不会损害生成质量？这是个尚未充分研究的交叉点。
3. **指令引导压缩的上限。** 把压缩和问题耦合是趋势，但它要求"压缩"发生在"看到问题之后"，与 prefill 阶段的预计算/缓存复用相冲突——如何兼顾两者是开放工程问题。
4. **统一评测的缺失。** 整个领域缺一个**跨任务、跨敏感度**的标准评测协议，导致论文间的"压缩比-性能"数字很难直接比较。

> **资源指针**：社区维护的 [Awesome-Token-Compress](https://github.com/daixiangzi/Awesome-Token-Compress) 列表持续跟踪 ViT/VLM 的 Token 压缩工作，是这个方向最好的入口。
