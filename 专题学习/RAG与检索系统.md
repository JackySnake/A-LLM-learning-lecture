## 一、为什么 LLM 需要外挂检索

模型参数里存的知识（**参数化知识**）有三个治不好的病：

1. **过时**：训练截止日之后的世界一无所知，且更新要重训；
2. **幻觉**：对长尾事实，模型倾向于"流畅地编"而不是"诚实地说不知道"——因为预训练目标奖励的是合理续写，不是事实核查；
3. **私有数据缺席**：你公司的文档、代码库、工单系统不在训练集里。

RAG（Retrieval-Augmented Generation）的思路：把知识从参数里搬出来，放进**非参数化存储**（文档库），生成时按需检索、塞进上下文。知识更新变成改文档（秒级、可审计），而非重训模型。

**先校准一个 2026 年的视角**：经典 RAG（固定管线"检索一次→生成"）作为产品形态确实在退潮，但**检索本身作为 Agent 的一个工具**地位反而上升了（见第五节 Agentic RAG）。学 RAG 不是学某个框架的管线搭建，而是学"如何把对的信息以对的成本送进上下文"——这个问题在 Week 9 记忆检索、Week 10 技能检索、Week 11 工具检索里反复出现，**它们在数学上是同一个问题**。

## 二、基础管线：四个环节，每个都有坑

```
文档 → ① 切块 Chunking → ② 向量化 Embedding → 入库
                                                  ↓
查询 → ② 向量化 → ③ 检索 Retrieval → ④ 重排 Rerank → 拼入 Prompt → 生成
```

### 2.1 切块（Chunking）：最廉价也最被低估的环节

核心矛盾：**块太小 → 语义不完整，检回来没法用；块太大 → 一个块里混多个主题，向量被"平均"得面目模糊，且浪费上下文**。

| 策略 | 做法 | 适用 |
|---|---|---|
| 固定长度 + 重叠 | 如 512 token、10–20% overlap（防止句子被拦腰切断） | 兜底默认 |
| 结构感知 | 按 Markdown 标题 / 函数边界 / 段落切 | 文档、代码（首选） |
| 语义切分 | 相邻句子 embedding 相似度骤降处断开 | 无结构长文 |
| Parent-Child | 用小块检索（精准定位），命中后返回其所属大块（上下文完整） | 通用增强，性价比高 |

一个 2024 年后的重要补丁——**Contextual Retrieval（Anthropic，2024-09）**：切块的根本伤害是**块失去了全局语境**（"该公司收入增长 3%"——哪家公司？哪个季度？）。解法简单粗暴：入库前用 LLM 给每个块生成一两句"这个块在原文中的位置和背景"前置说明，再做 embedding。Anthropic 报告检索失败率下降 ~49%（叠加 reranking 后 ~67%）。成本靠 prompt caching 压低（整篇文档作为缓存前缀，逐块生成说明）——这是 Week 6 推理优化知识反哺数据工程的好例子。

### 2.2 向量化（Embedding）：双塔的能力与天花板

Embedding 模型把文本映射到向量空间，使"语义相近 ≈ 距离近"。检索时用**双塔（bi-encoder）**模式：文档向量离线算好，查询向量在线算，算一次点积/余弦相似度即得分。

- **选型现状（2025–2026）**：开源以 BGE-M3、GTE、**Qwen3-Embedding** 系列为主流（MTEB 榜单是事实标准）；闭源有 OpenAI text-embedding-3、Voyage（2025 年被 Anthropic 收购，可见检索对 Agent 厂商的战略价值）。多数支持 instruction-aware（查询前加任务指令）和 Matryoshka（同一向量可截断成短向量用，精度换成本）。
- **天花板要心里有数**：单向量是对整段文本的**有损压缩**——查询与文档的细粒度词级对应关系（谁修饰谁、否定词管到哪）在压缩中丢失。这是后面 rerank 和 late interaction 存在的根本原因。

### 2.3 检索（Retrieval）：ANN 与混合检索

**ANN 索引**：库大了不能暴力扫描，用近似最近邻。主流是 **HNSW**（分层可导航小世界图）——直觉：建一座"高速公路分层"的图，顶层稀疏长边快速逼近目标区域，底层稠密短边精确定位；查询复杂度近似 $O(\log N)$，代价是内存和召回率的轻微损失。Week 9 A.3 提到的各向量库（Chroma/Milvus/pgvector 等）底层多是 HNSW 或其变体。

**混合检索（Hybrid Search）——工程上几乎总是该开的开关**：

- 向量检索擅长语义改写（"怎么退钱" ↔ "退款流程"），但对**精确字符串**不可靠：产品型号、错误码、人名、函数名这类 token 在向量空间里没有特殊地位；
- **BM25**（词频统计的经典算法）正好相反：精确匹配极强，语义泛化为零；
- 两路各自检索，用 **RRF（Reciprocal Rank Fusion）**融合：$\text{score}(d) = \sum_r \frac{1}{k + \text{rank}_r(d)}$——只看排名不看分值，天然解决"两路分数不可比"的问题（k 通常取 60）。

### 2.4 重排（Rerank）：用算力换精度的关键一跳

为什么检索完还要排一次？回到双塔的天花板：查询和文档**各自独立**编码，模型从未"同时看到"两者。**Cross-encoder reranker** 把查询和候选文档拼在一起过一遍 Transformer，注意力可以在两者的 token 之间交叉流动，精度显著高于双塔——但每对都要前向一次，只能用在小候选集上。

于是形成标准的**两级漏斗**：

$$\text{百万级文档} \xrightarrow{\text{双塔+BM25（快而糙）}} \text{Top 100} \xrightarrow{\text{cross-encoder（慢而准）}} \text{Top 5–10 进上下文}$$

这个"召回-精排"结构与推荐系统完全同构。理解它的本质（**交互越早、精度越高、成本越高**）比记住任何具体模型名都重要——它也解释了 late interaction（见第六节）为什么是介于两者之间的折中。

## 三、查询侧优化：检索效果差，往往是问题没问对

文档侧做得再好，查询本身烂也白搭。三个常用手段：

1. **查询改写（Multi-Query）**：让 LLM 把原查询改写成多个角度的变体，各自检索后合并——对抗"用户措辞与文档措辞不重叠"；
2. **HyDE**：让 LLM 先**凭空写一个假想答案**，用假想答案（而非问题）去检索——直觉：答案与文档的向量距离比"问题与文档"更近（问题和答案在语义空间里本来就不是一类东西）；
3. **查询分解**：复杂问题拆成子问题分别检索（"对比 A 和 B 的政策" → 检 A、检 B）——这已经一只脚踏进 Agentic RAG 的门里了。

## 四、评估：没有度量就没有迭代

RAG 是两段系统，**必须分开归因**——答案错了，是没检到（检索问题）还是检到了没用好（生成问题）？

**检索侧**（需要标注的相关性数据）：
- **Recall@k**：相关文档有没有进 Top-k——漏斗第一级只看这个（精度交给 rerank）；
- **MRR / NDCG**：排名质量——相关文档排得越靠前越好，NDCG 额外支持分级相关性。

**端到端**（RAGAS 框架定义的四象限，用 LLM-as-Judge 计算，与 Week 11 评估体系方法论相通）：

| 指标 | 问的问题 | 归因 |
|---|---|---|
| Context Recall | 答案需要的信息，上下文里有吗 | 检索 |
| Context Precision | 上下文里有多少是相关的（噪声多吗） | 检索 |
| **Faithfulness** | 答案的每个断言都有上下文支撑吗 | 生成（幻觉检测）|
| Answer Relevance | 答案对题吗 | 生成 |

工程惯例：先用 ~50 条人工核验的"黄金问答集"把检索侧调到 Recall@10 ≥ 90%，再去折腾生成侧——顺序反了就是在噪声上调参。

## 五、Agentic RAG 与"RAG 已死"之争

### 5.1 长上下文 vs RAG：一场被标题党耽误的讨论

"上下文窗口都 1M 了，还要 RAG 干嘛？"——2024 年起反复出现的论调。三个事实让它站不住：

1. **Context Rot**（Week 9 4.4）：窗口装得下 ≠ 注意力顾得过来，塞满长上下文的检索精度随长度劣化；
2. **成本**：每次请求都把全库塞进 prompt，token 费用和延迟是检索的几十上百倍（prefix caching 可缓解但不治本）；
3. **库的规模**：企业文档库轻松上 GB，物理上不存在装得下的窗口。

正确的合题：**长上下文改变的是检索的"粒度"而非"必要性"**——从前检索 512 token 的碎块，如今可以检索整篇文档甚至整个目录（粗检索 + 长上下文精读），漏斗变粗了，但漏斗还在。

### 5.2 Agentic RAG：从固定管线到检索策略

经典 RAG 的硬伤是"**检索一次、不管好坏、直接生成**"。Agentic RAG 把检索变成 Agent 的一个工具，把"检索策略"交给模型决策：

- 要不要检索？（简单问题直接答，省一次检索）
- 检索结果够不够？不够就**换关键词再检**、分解子问题、或换数据源；
- 检到的内容互相矛盾？交叉验证。

这正是 Week 9 ReAct 循环的检索特化版，也是 Deep Research 类产品的骨架。更进一步——**检索策略本身可以被 RL 训练出来**（何时搜、搜什么、搜几轮），这就是 [AgenticRL 专题](AgenticRL与Agent训练.md) 战线二（Search-R1 等）的主题。三个层次串起来看：

$$\text{固定管线（流程写死）} \to \text{Agentic RAG（prompt 驱动决策）} \to \text{Search RL（决策训练进权重）}$$

### 5.3 GraphRAG：当问题是"全局的"

向量检索回答**局部事实**问题很好（"X 的退款政策是什么"），但对**全局综合**问题失效（"这批文档的主要争议主题有哪些"）——答案不在任何一个块里。微软 GraphRAG（2024）的方案：离线用 LLM 从全库抽取实体-关系图，对图做社区检测并逐层生成社区摘要；全局问题查摘要层，局部问题仍走向量。代价是索引构建的 LLM 成本极高，属于"知道何时该用"的武器而非默认选项。

## 六、前沿动态（2025–2026）

- **Late Interaction / ColBERT 系**：介于双塔与 cross-encoder 之间的第三条路——文档存**每个 token 的向量**（而非压成单向量），查询时做轻量的 MaxSim 词级交互。精度接近 cross-encoder，速度接近双塔，代价是存储膨胀。**ColPali / ColQwen**（2024–2025）把这套思路用到文档**截图**上：跳过 OCR/解析，直接对页面图像做多向量检索——表格、图表混排的 PDF 检索效果显著超过文本管线，是多模态 RAG 当前的主流答案（与 Week 8 视觉编码知识衔接）。
- **Anthropic**：Contextual Retrieval（2024）之后，2025 年收购 Voyage AI 补齐 embedding/rerank 自研能力；Claude 产品线的检索基建（Projects、Agent 的文件搜索）持续下沉为平台能力。
- **OpenAI / Google**：File Search（托管 RAG）走"检索即服务"路线——管线各环节被平台收编，应用方只管传文件；Gemini 主打长上下文 + 检索混合。
- **开源生态**：LlamaIndex / LangChain 的管线抽象逐渐让位于更轻的"检索工具 + Agent 循环"组合（Agentic RAG 趋势的框架侧反映）；评估侧 RAGAS 成为事实标准。
- **研究热点**：检索与推理的联合训练（Search RL，见 AgenticRL 专题）、记忆与 RAG 的融合（Week 9 A.5 的 A-MEM 这类"会自己长结构的库"）、多模态统一检索（文本/图像/表格一个索引）。
